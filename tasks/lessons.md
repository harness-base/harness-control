# 错题本 lessons

每踩一次坑记一条，三段式，按日期倒序（新的在上）。反复出现的，升级进对应 skill 或 rule（rule-0007）。

格式：

```
## YYYY-MM-DD：一句话标题 <!-- opt: seen -->
- Mistake：错在哪
- Prevention：以后怎么防
- Earlier signal：怎么更早发现
```

**整理标记**（晋升流程 / step 4 用，标在标题行尾，`scripts/lessons-promote-check.sh` 据此计数）：无标记 = 未整理 ｜ `<!-- opt: seen -->` = 提醒过·待你决定 ｜ `<!-- opt: skip -->` = 看过·不升 ｜ `<!-- opt: rule-00NN -->` = **已升成该编号规则，或已被该编号既有规则覆盖、不另升**（两种来路读正文分辨：正文明说"被既有规则管着 / 同款 / 既有精神"即覆盖型，非晋升源）。攒够未整理的，钩子会提醒整理（升规则走 `hc-add-rule`）。

## 2026-07-13：根 README「接入一个工程」还在教 hc-onboard 之前的手工三步——门面文档不在 sweep 名单，用户逮的
- Mistake：ADR-0017/0018 把接入升级成 hc-onboard 引导式时，PROJECT_ONBOARDING.md 重写对了，但根 README:63"手工三步即可纳管"没跟——门面入口教旧走法一直到用户按 README 准备接老项目才发现。断环：doc-sync-checklist 只有"顶层目录变→README 目录表"行，没有"流程被 skill 承接→README 叙述节"行；对抗/eval 只盯批内连带清单，README 不在那两个 ADR 的清单里。
- Prevention：机制换走法（手工步骤→skill 承接、旧闸→新闸）时，sweep 名单必含**门面文档的叙述节**（根 README、docs/README）——已补 doc-sync-checklist 一行兜底。
- Earlier signal：README 里有"即可 / 只需 N 步"式手工流程叙述、而同主题已有 skill 承接——两者并存就是漂移，grep "即可" 这类词能扫出来。

## 2026-07-10：todo 项写着 PROJECT_ONBOARDING 打了 [x]，实际那文件一个字没动——打勾前没对 git status <!-- opt: rule-0003 -->
- Mistake：契约对照批 todo 第 10 项列了"…+ onboard-reviewer 双栈判据④枚举 + PROJECT_ONBOARDING）"，我做的时候只改了 reviewer 标题句、PROJECT_ONBOARDING.md 压根没碰，却把整项勾成 [x]——对抗验证拿 git status 对账当场戳穿（该文件不在改动清单里）。rule-0003"不许假完成"的 todo 变种：勾选也是"声称"，声称就要有实改证据。同型链：上两批是"修复轮不回填 ADR"，这批升级成"首轮就没做但打了勾"。
- Prevention：勾任何含多个子项的 todo 项之前，把项里点名的**每个文件**对一遍 `git status`——名单里有、改动里没有 = 不许勾（要么补做要么拆项标未完成）。ADR 受影响栏/连带清单同款（已有 lesson），todo 勾选从此并入同一个动作：**收尾自检 = 名单 ↔ git status 对账**。
- Earlier signal：一个 todo 项里点名 ≥3 个文件时，勾选前的对账几乎必有漏——点名多的项拆开勾，别整项一把勾。

## 2026-07-09：为回应"会不会太重"临场发明"脚本先 diff"，被戳穿格式根本不同——别为安抚担心加不成立的机制 <!-- opt: skip -->
- Mistake：用户问"routelist 全量返回、agent 对比会不会太重"，我为了显得轻，临场加了"让脚本先做集合差、agent 只看 diff 结果"——被用户戳穿：契约（markdown 端点表格）和 routelist（纯文本行）**不是一个格式**，没法机械 diff；真要 diff 得先写解析 markdown 的脚本，正是仓里栽过的裸 grep 解析 markdown 坑（lessons 2026-06-28）。而正确答案本来很简单：几百行以内的清单对比 agent 本来就不重，且语义对齐（`:id` vs `{id}` 写法差异）正是 agent 比脚本强的地方——用户最初定的"agent 来判断对比"就是对的。
- Prevention：用户表达"会不会太重/太慢"类担心时，先**核担心是否成立**（量级多大、agent 处理这个量级吃不吃力），成立才谈优化；不成立就直说"这个量级不重、因为 X"。别用临场发明的机制安抚——新机制自己要过"格式成立吗/引入什么坑"的检查。
- Earlier signal：为回应担心而**新加**的机制若 10 秒内想出来的，先自问"这两个东西真能这么接吗"——格式/接口/前提对不上就是拍脑袋。

## 2026-07-09：设计"取数"环节默认让 agent 临场提取，用户纠正为脚本化——取数用脚本、判断用 agent <!-- opt: skip -->
- Mistake：契约对照场景我设计成"agent 按优先级自己提取 routelist（含读路由注册代码兜底）"。用户纠正：这该是**偏脚本验证的场景**——路由导出由**脚本**实现（接入 verification.yaml 的 script 体系、onboard 时创建），agent 只做"拿脚本输出和契约**对比判断**"；"agent 读代码提取"这类临场取数被砍。背后偏好：**确定性的取数环节脚本化**（可重复、可审计、不随 agent 发挥漂），agent 留给语义判断——与两层防线（机器管结构/agent 管判断）同构。
- Prevention：设计流程时凡遇"从工程取一份事实清单"（路由/schema/依赖/配置项……）的环节，默认**问一句"这该不该是工程接入点脚本"**（verification.yaml 登记、onboard 建、三态占位），而不是让 agent 每次临场扒代码；agent 的角色放在"拿脚本输出做语义对比/归因"。
- Earlier signal：方案里出现"agent grep/读代码提取 XX 清单（可能漏）"——"可能漏"三个字就是信号：取数不确定 = 该下沉成脚本。
- 处置注（2026-07-10）："取数用脚本、判断用 agent"已成文 ADR-0026（可复用模式），不另升规则。

## 2026-07-09：对抗修复轮改了新文件，ADR 受影响栏/连带清单不回填——同型第二次被 eval 逮 <!-- opt: rule-0007 -->
- Mistake：机制体检批对抗验证逮出 8 major，修复时改了 hc-git-workflow 分级表、scripts/README、hc-eval 四件套等——但 ADR-0025 受影响栏还是 build 时写的旧账（hc-git-workflow 标"否"、连带清单漏 6 个实改文件）。上批（features 退役）eval 也逮过同款"巡查后补改的 ~8 处不在 ADR 连带清单"。两次同根：把 ADR 受影响栏当"build 时写一次"的静态文档，忘了它要**随修复轮滚动回填**。
- Prevention：每轮对抗/eval 修复涉及**新文件**时，修完顺手把 ADR 受影响栏/连带清单对一遍 git status 回填——固定动作：修复 commit 前 `git status` 对 ADR 连带清单，多改的补进去、声称"否"的复核。
- Earlier signal：修复清单里出现"改某 skill/某文档"而那文件不在 ADR 连带清单——补修复时就该同步补账，不等评委抓。
- 处置注（2026-07-10）：受影响栏随修复轮回填是 rule-0007（ADR 记录受影响面）的执行纪律、被其覆盖不另升；固定动作已沉淀进 `docs/harness/adversarial-review.md` 核心 pattern 第 4 步（"声称清单 ↔ git status"对账，与 todo 打勾同口径）。

## 2026-07-09：提"L0-L6 收敛三档"还是档位思维，没看出"读多少"半边已被渐进式引用结构整体取代 <!-- opt: skip -->
- Mistake：用户问 L2 分级要不要优化，我提"收敛成三档、双职责（定读多少+定 eval 门槛）"。用户点破："读多少还需要定吗？已经是渐进式引用了，AI 自己判断读什么就行；只需要一个标准判断 eval。"——加载档位表是 2026-06 早期设计，当时还没有就近 AGENTS.md/目录 README 规则/skill references 分档/分线文件"做哪线读哪份"；这套渐进披露结构建成后，"先判 L 几再决定读什么"实际没人做（我自己干活也是顺引用链走）。档位的唯一实质残余 = eval 门槛。我修补旧机制而不是识别它已被结构取代——与 features 退役同款（用户连续第二次逮到）。
- Prevention：给老机制提优化方案前，先问"它的职责现在被什么结构承载着"——若新结构已整体覆盖（引用链/就近加载/按需分档都在干"读多少"的活），方案该是**退役+保留唯一实质用途**，不是瘦身。harness 里"机制被结构取代"的信号：机制要求的动作（判档位）在实际工作流里从不显式发生。
- Earlier signal：我自己从没在任何任务开头真的"判定 L 几然后照表读"——自己都不执行的机制，就是该退役的机制。

## 2026-07-08：把"历史文档不改写"当处理完毕，漏了"旧文档读者怎么知道口径变了"——不改写 ≠ 不导航 <!-- opt: skip -->
- Mistake：features 退役批我定了"历史 ADR 正文不改写、以 ADR-0023 为准"就当处理完。用户追问"features 被引用的地方不处理吗？你得考虑啊"戳破：读旧 ADR（如 0003 定的"PRD 可派生 feature 包"）的人**不会先读 0023**，没有任何路标告诉他这段已被推翻——"以新 ADR 为准"只对已经知道新 ADR 的人有效。而本仓有现成先例我没跟：ADR-0009 被 0021 演进时，头部加了"> **更新（日期）**：已由 ADR-0021 演进…下方叙述保留历史原样"指针块。巡查（盯活文档残留）和 eval（盯声称真实性）都没抓这层——"历史文档缺前向指针"不在任何一层的判据里。
- Prevention：退役/推翻一个机制时，历史文档处理是**两件事**：①正文不改写（历史是真的）②**决策被实质推翻的旧 ADR 头部加前向指针块**（related_docs 补新 ADR）。判据："该 ADR 确立的决策有一部分被本次推翻"才加（顺带叙述性提及不加，防指针稀释）。这条该进 hc-add-rule 对照表或 ADR 模板说明。
- Earlier signal：写"以本 ADR 为准"这句话时就该自问——"读旧文档的人怎么到达本 ADR？"没有入口路径 = 这句话只是自我安慰。

## 2026-07-08：给 features 退役提方案时只给了"换锚保硬门禁"两选项，漏了用户的一贯哲学"松耦合、提示不强耦" <!-- opt: skip -->
- Mistake：rule-0001 退役方案我给的两个选项都是"把 MUST STOP 门禁换个锚定物继续硬卡"（approved 故事 / 完整 PRD）。用户拍的是第三条路：**没有任何门槛**——各 skill 松耦合、不强依赖其他 skill 的产物、可独立工作；关联 skill 里只加**提示**（指路上游）不做硬耦合；唯一的"整体"是 hc-dev 自身走 superpowers 先 plan。而"松耦合、可跳过、只提醒不强制"本就是仓里反复出现的哲学（ADR-0003 松耦合、testing-flow"每步解耦用户可跳过任意一步"、hc-prd→hc-test 软提醒）——我提案时没把这个既有哲学当默认框架。
- Prevention：给 harness 机制提方案时，先回想这仓的架构哲学再列选项——**skill 间默认松耦合（提示/指路），硬门禁是例外**（只留给真红线如密钥/假完成）；涉及"要不要强制"的选项，必须包含"不强制、只提示"这一档，且多半该是推荐项。
- Earlier signal：我列的所有选项都含"MUST STOP/硬闸"而用户过往从没主动要求过新增硬门禁（反而砍过"每次检查太蠢"的钩子）——选项集的强制度分布和用户历史偏好背离，就是框架错了。

## 2026-07-08：判"features 体系没废弃"只看引用还挂着，没看实质工作流早已绕开它——形式活着 ≠ 实质活着 <!-- opt: skip -->
- Mistake：用户问"feature 目录是不是废弃了"，我 grep 到 rule-0001/hc-dev 门禁/index-audit 还指着它就答"没废弃、在休眠"。用户纠正：F-0001~0006 那套已经没有了、feature 包承载的东西被打散进新环节（需求→prds、设计→designs、测试→test-cases），旧的"一个包走到底"workflow 已不存在，hc-dev 实际不需要 feature 包。我犯的错：把"文档里还有引用"当成"机制还活着"——引用是**惯性残留**，实质判据应该是"现在的工作流里，这个环节还产生/消费它吗"。
- Prevention：判一个机制"废没废"用双判据：①引用还在吗（形式）②**当前主工作流还流经它吗**（实质）——新链条建成后旧机制引用未清 = 不是"活着"而是"待退役的漂移"。答"废没废"这类体检题，先画当前主链条（谁产谁消），再看目标机制在不在链上。
- Earlier signal：一个目录 initial commit 之后零新增、而同期新体系（prds/designs）在持续长——这就是"被取代"的信号，不是"休眠"的信号；我自己都注意到了"六周零新增"却选择了保守解读。

## 2026-07-08：让用户拍决策时上下文没铺够，连问两轮都被打回（"没懂"→"没头没尾我怎么决策"） <!-- opt: skip -->
- Mistake：review 发现 lessons marker 语义错用，要用户拍修法。第一轮问题里塞满内部术语（opt: rule-00NN / 词表 / hc-rule-reviewer）→ 用户"没懂"；第二轮换了比喻但只解释了"错在哪"，没讲**这套标签机制是干嘛的、谁在消费它、两个修法各影响什么下游**→ 用户"你不能描述清楚上下文再问吗？没头没尾我怎么决策"。同根前科：黑话 lesson、AskUserQuestion 堆规则号 lesson——但这次的新根因是**决策请求缺上下文链**：用户拍板需要完整因果（机制是什么→怎么用的→哪坏了→修法影响什么），我只给了中间一段。
- Prevention：让用户拍任何决策前，自查四件事讲全没：① 这个机制/东西是干嘛的（一句白话）② 这次为什么牵扯到它（发生了什么）③ 坏在哪、不修的后果 ④ 每个选项动什么、影响什么下游。四缺一就先补再问。宁可正文长一点，选项本身反而要短。
- Earlier signal：起草的问题里出现"仓内专名 + 用户最近没接触过的机制"却没配一段"这是干嘛的"——就是没头没尾的信号；换比喻不等于补上下文，比喻只替代①、替代不了②③④。
- 处置注（2026-07-10）：四件套自查已沉淀进跨会话 memory（ask-with-recommendation 条目），不升规则（沟通纪律非任务红线）。

## 2026-07-08：把 harness 文档资产的 review 当代码 review 套满 10 角度 + 每 candidate 一 verifier，被用户双重叫停 <!-- opt: skip -->
- Mistake：用户说"review 一遍"，我直接套 code-review skill 的 xhigh 全家桶——10 个 finder 角度（含语言陷阱、wrapper/proxy、efficiency 等**为可执行代码设计的角度**）+ 每 candidate 一个 verifier + sweep，可能 50+ agent。用户纠正两点：① harness 工程的 review 是**抽象的语义审**——重点只有"遗漏的关联项 / 语义冲突 / 前后矛盾"，没有严谨语法逻辑可挖，别当代码 review；② 这个量级用不了这么多 agents。
- Prevention：review 对象是**文档/控制面资产**时，别套 code-review 的角度矩阵——就 2-3 个对口视角：关联项遗漏（现成机制 = hc-doc-sync-reviewer 对照 doc-sync-checklist）、语义冲突/前后矛盾（跨文件口径对照）、必要时指针有效性；编制 2-3 个 agent 封顶。code-review skill 的 effort 分级是给代码的，文档审的"深"不等于"角度多"，等于"对照面全"。
- Earlier signal：起 review 前问一句"这个 diff 有没有可执行代码"——全是 .md/.toml/.yaml 时，语言陷阱/wrapper/efficiency 角度必然空转，就该换文档审的编制。
- 处置注（2026-07-10）：已沉淀进 `docs/harness/adversarial-review.md`「按对象类型伸缩」（文档/控制面资产审 2–3 对口视角封顶），不升规则。

## 2026-07-08：往已合并 PR 的分支追加 commit → 成孤儿（既不在 merged PR、也不在 main） <!-- opt: skip -->
- Mistake：hc-onboard 拆分 commit（d2e5131）push 到 feat 分支时，PR #15 其实已在前一个 commit（a20ecef）合并了——合并点在 d2e5131 之前，导致拆分 commit 成孤儿：不在 merged #15、也不在 main。开下一个 PR 时 `git merge-base --is-ancestor` 才发现它 1 commit ahead of main、会连带进新 PR。
- Prevention：**PR 一旦 merged，它的分支就"关了"**——再往该分支 push 的 commit 不会自动进 main。给分支追加 commit 前先 `gh pr view <branch> --json state`；若 MERGED，新工作**从最新 main 切新分支**，别在已合并分支上续。发布起手式的"核目标 PR 状态"要延伸到"push 完确认 commit 真进了 PR 的 head / 会被合并"——**push 成功 ≠ 进了 PR**。
- Earlier signal：push 输出显示 `a20ecef..d2e5131` 更新了分支，我就当"进 PR 了"——没回头核 PR 是否还 open、这 commit 会不会被合。分支更新与 PR 合并是两件事。

## 2026-07-08：用户问"能不能给 Codex 仿 workflow"，我却跳去 brainstorm"review 要多广多深"——抓错了层（原语 vs 政策） <!-- opt: skip -->
- Mistake：用户说"让 review subagent 开 workflow 模式做对抗评审"，我直接进 brainstorm 问"广度/深度/通用"三选一（review 政策）。用户"其实我是想，在 Claude Code 用 workflows 模式，Codex 里没有类似模式，可以仿照这个模式吗、能实现吗"点破：他真正问的是 **Claude Code 有 Workflow 编排原语、Codex 没对等的，能不能仿一个**——是**跨运行时双栈能力对等**问题，不是 review 该多严的政策问题。我把"用什么原语实现"错当成"review 要多深"。
- Prevention：用户提到某"模式/能力"时先分层——他问的是①这个能力本身能不能有/怎么来（原语层）还是②拿它做什么（政策层）？出现"Claude 有 X、Codex 没有、能不能仿"这种话 = 双栈对等题，先摸 X 是什么原语、两边各支持什么，别跳去设计用 X 干的活。

- Earlier signal：在一个新 sub-task 里要 git commit/push，而本 sub-task 用户没单独说过"提交 / 推"——就该停下确认。

## 2026-06-29："执行吧"后没有决策点，我却造了个 checkpoint 停下来问"继续还是暂停" <!-- opt: skip -->
- Mistake：用户说"执行吧"、设计已逐条敲定（无悬而未决的决策点），我做了 T1/T2 + T3 一部分后，出于自己的谨慎 / 怕出错，manufactured 一个"自然停点"停下问"现在做完还是改天"。用户："谁让你停的？没决策点为什么停下来问？"——这不是真 checkpoint，是我替用户做了"要不要继续"的决定，而那本不该问。

- Earlier signal：在写一个 harness 改动的 spec/plan，却没在开头说"用 self-evolution"——入口就漏了。

## 2026-06-29：反复给"单场景补丁"，用户要的是"为什么 agent 根本没意识到要改关联项"的根 <!-- opt: rule-0011 -->

## 2026-06-29：把 turn-backstop 的三条触发压成"计数器"，还在这个错简称上做设计 <!-- opt: rule-0008 -->
- Mistake：turn-backstop 是**三条 OR 触发**（K 轮 / commit 边界 / 变更文件数增量，`turn-backstop.sh:47-51`）。我第一次解释时说对了，但后两轮顺嘴压成"计数器闸"，还拿这个 lossy 简称当设计依据（论证"只用钩子"）。用户纠正"不止计数器、还有别的触发"。

## 2026-06-29：已有计数器钩子兜底，我还要"收尾也每次主动跑"，多此一举 <!-- opt: skip -->

## 2026-06-29：把"夹带的数据被机器读"当成"该是 skill"的理由，给 doc-sync 判了软 keep <!-- opt: skip -->
- Mistake：我判 doc-sync "borderline keep"，理由是"它那张 checklist 被 turn-backstop 钩子读、load-bearing"。用户反驳：如果有用的只是那张表（数据），就**不该是 skill**——数据该独立成数据、检查该写个 subagent 干。我把"文件里夹带了一段被机器读的数据"错当成"配当 skill"的理由；而 skill 该挣的是"具体触发 + 主动走的流程 + 产物/闸"，不是"我夹带了一段有用数据"。还顺带漏看了 optimization-log 闭环根本没在跑（真踩的 README 漂移从没被捞到）。

- Earlier signal：替一个 skill 辩护时，理由落在"它文件里某段数据有用"而不是"这流程有人会主动走、产出了什么"——就是在替壳找借口。

## 2026-06-29：用户说"我忘了 X"之后，我还用简称（"那张表"）指 X 的内部，把人绕晕 <!-- opt: skip -->

- Earlier signal：同一轮里用户连着问"fan-out 是什么""怎么成表了"——连续追问我的用词，就是我在甩没锚定的简称、对方已跟不上。

## 2026-06-29：按名字把 skill 预归类成"降级候选"，没读就下判断 <!-- opt: skip --><!-- 原标 rule-0004，该规则已退役（ADR-0025）；坑（没读就下判断）由 rule-0008 不假设精神与 hc-self-evolution 流程覆盖，改 skip。 -->

## 2026-06-29：往 README 加内容时只顾加自己那段，没顺手核对既有内容已漂移 <!-- opt: rule-0012 -->

## 2026-06-29：并行的两个 worker 共享一套可变命名空间（都造 FP 号）→ 必撞号 <!-- opt: skip -->
- Mistake：prd 编排把 PRD本体员 + 功能点员 + 原型员 设成**纯并行**，而 PRD本体员和功能点员**都会造 `FP-NN` 编号**。并行时俩看不到对方、各编各的 → 同号指不同物（`FP-08` 一份是"桌面通知"、另一份是"权限降级"）、跨文档追溯断裂。**end-to-end dogfood 才暴露**——`make verify` 的结构检查查不出这种语义级撞号。

## 2026-06-29：把"dogfood"用成了拿旧 code-reviewer 审代码，新建的 prd-reviewer 一次没跑过 <!-- opt: rule-0003 -->

## 2026-06-29：切任务时没清上一任务的 `## Review` 段，stop-check 拿旧 Review 误判新任务"在收尾" <!-- opt: rule-0013 -->
- Mistake：开 prd-orchestration 新任务时，我只改了 `todo.md` 顶部的"当前任务"头，**把上一任务（dev-skill）整段连同它的 `## Review` 留在了文件里**。stop-check（rule-0005 收尾闸）按"当前 `## Review` 存在 = 在收尾"判定，撞到那段旧 Review，就在我**任务中途**（T6 还在跑、eval 还没到）⛔ 拦截，要我先跑 eval。是 todo 卫生没跟上，不是闸的 bug。

## 2026-06-29：每条消息结尾"认吗/认不认"问个不停，用户嫌"像审犯人认罪" <!-- opt: skip -->

## 2026-06-29：把"交互对话"当成不能拆 subagent，漏了"需求采集"本身也是一个 subagent <!-- opt: skip -->
- Mistake：聊 PRD 编排时我说"对话留人、不 fan-out"，把"跟用户多轮聊需求"当成必须留主 agent、不可编排的一步。用户纠正：需求采集**本身也是一个 subagent**（需求采集员），职责就是跟用户对话；总监（编排 agent）照样调度它。用户："刚刚我不说，你就不提醒"——我框窄了。

## 2026-06-23：弹性 builder 忽略调用方 ctx，把"热更落地"的证据搞成脆弱的超时巧合 <!-- opt: rule-0009 -->
- Mistake：`redisx.Open`/`pgxpool.Open` 连通性探测自建 `context.Background()+DialTimeout`，无视调用方 ctx。后果有二：①热更到坏地址时 `Provider.Build` 里的 eager ping 阻塞、不被 readyz 请求 ctx 取消（kratos HTTP 默认请求超时 1s），readyz 卡满 1s 再报 `context deadline exceeded`；②AC-CR1 "坏 redis 地址→/readyz 翻 503" 之所以成立，靠的竟是"Open 阻塞耗光那 1s → self-heal 回退旧好句柄后 Health 再探时 ctx 已过期"这个**超时巧合**，而非干净的"provider 改连坏地址、探活失败"。读码审计据此（误）判它必为 200 假阳性；实跑确是 503，但机制脆弱、与脚本注释不符。
- Prevention：弹性数据面 builder 一律**吃调用方 ctx**（`Open(ctx, cfg)`，ping ctx = min(调用方 deadline, DialTimeout)）。**别让断言依赖看不见的 ctx 超时竞态**——S6 给 readyz 设 15s 合理探测超时后，"坏 redis 地址→503"当场失效（self-heal 回退旧好句柄、readyz 保持 200，这才是对的弹性）。最终把"热更续上"改成 ctx 无关的硬证据：恢复后推非法配置 → confcenter 新增 `retaining previous config` 日志（BEFORE/AFTER 计数对比，杜绝旧行假命中）。教训：一个断言若随"超时设多少"而翻盘，它锚的就不是真行为。
- Earlier signal：readyz 失败信息是 `context deadline exceeded`（超时）而非 `connection refused`（干净的不可达）——错误类型不对就该追：谁在等谁的 ctx。
