# 引导式把工程接进 harness（hc-onboard）

本 skill 管"**把一个工程按 harness 规范搭好、接进控制面**"——**新项目**：从零起挂进 `projects/`，配好最小骨架、决策记录、执行口接线（依据 **ADR-0017**）；**老项目**：已有代码 + 历史的工程**倒着对齐**，把项目原有规范 / 老资产扫→确认→搬进 harness 规矩（依据 **ADR-0018**）。两条岔路过审后都交棒给开发链条（`hc-prd` → `hc-tech-design` → `hc-dev`）。

**形态 = 交互式引导**（主 agent 当**接入向导**），同 `hc-tech-design` 的「引导 → 用户确认 → 派 reviewer 对抗挑刺 → 回改 loop」——一条连贯的接入对话、逐步确认、不可拆并行。

**铁律（贯穿全程）**：选型 / 结构 / 规矩都是"**选择**"——**摆选项 + 讲取舍让用户拍板，不替用户定、不"看见啥落啥"**。接入向导的活是把路铺清楚、把取舍讲明白，拍板权归用户。

## ① 何时用 / 何时不用
- **用**：把**一个工程接进 harness**——从零起一个新项目（配项目名 / 栈 / 目标 → 搭最小骨架 → 记第一个决策 ADR → 接执行口 → 过审 → 登记），或接管一个**已有代码 + 历史的老工程**（拆模块 → 按模块 扫→确认→搬进规范 → 接执行口发现+对齐 → 引入关联进主目录）。用户说「接入项目 / 新建项目 / 挂个工程 / onboard / 起个新工程 / 接入老项目 / 存量项目 / 对齐现有项目」时用。
- **不用**：**已接工程内**写功能 / 改 bug / 重构走 `hc-dev`；产**需求**走 `hc-prd`；产**研发方案 / 接口设计**走 `hc-tech-design`；产**测试用例**走 `hc-test`；固化一条**规则**走 `hc-add-rule`；改 **harness 本身 / 查 harness 漏洞**走 `hc-self-evolution`。
- 一句话边界：`hc-onboard` 只**把工程接进来、搭壳、接线**，**不搭代码结构、不写实现、不产需求 / 方案 / 用例**——那些是接进来之后的下游链条的活（交棒见 ③ 第 7 步 / ④ 第 8 步）。

## ② 入口分流（先问 新 / 老，两分支都已实现）
接入向导第一句先分流，别急着往下走：

- **先问**：这是**从零起的新项目**，还是**已有代码的老工程要挂进来**？
- **新项目** → 走 ③ 的 **7 步**（从零搭最小骨架）。
- **老项目** → 走 ④ 的 **8 步**（有代码有历史，**倒着对齐**：把项目原有规范 / 老资产**扫 → 确认 → 搬进规范**，让"控制面记的 ↔ 项目真实代码"对得上、不漂）。同一个 skill 的第二条岔路：接执行口 / 评审 / 收尾 / 交棒的骨架与新项目共用，但取向不同——**别拿新项目 7 步硬套老工程**（老工程有存量代码 / 历史结构 / 隐性约定），也别拿老项目判据误伤新项目骨架。

## ③ 新项目 7 步（每步：问什么 / 确认什么 / 落什么）
一条连贯的引导对话，主 agent 当接入向导，**每步先确认再落**。每步都守铁律（摆选项 + 讲取舍，用户拍板）。

### 第 1 步 · 收基本信息
- **问什么**：项目名（**kebab-case**，将来是 `projects/<名>/` 目录名）/ 目标（这工程要解决什么）/ 概述（一句话它是什么）/ 技术栈（框架 · 语言）。**怎么问由 agent 临场把握**——用户一次给全就不必逐条追；给得零碎 / 含糊就补问，别拿"已有信息 + 想要的结果"硬编（rule-0008）。
- **确认什么**：把收齐的四项**复述给用户确认**——尤其项目名是不是想要的 kebab、栈选型用户是否认可（栈有取舍就摆出来让用户拍，别默认替他选）。**项目名必须跟现有工程不撞**：`projects/` 是**多工程**目录（可能已挂别的工程），动手前先核 `projects/` 现有目录名 + `workspace/verification.yaml` 现有 `name`，撞了让用户另起一个唯一的 kebab 名，绝不覆盖 / 混进现有工程。
- **落什么**：**先不落文件**，把确认过的信息作为后续步骤的输入。

### 第 2 步 · 搭最小骨架（只搭壳、不搭代码结构）
- **问什么**：这一步主要是"搭"不是"问"，但**红线 / 指针要不要项目专属项**得跟用户核（如这项目有没有额外红线）。
- **确认什么**：给用户看将要建的骨架清单，确认了再写。
- **落什么**：
  - `projects/<名>/` 目录；
  - 精简 **`AGENTS.md`**——只放**项目红线 / 技术栈 / 指针**（指向控制面根 `AGENTS.md` 与产物区），**不复刻**全局规则、不堆流水账；
  - 同级 **`CLAUDE.md` shim**（`@AGENTS.md`，让 Claude Code 就近发现）。
- **红线（本步命根）**：**只搭壳、不搭代码结构**。目录布局 / 分层 / 包结构 / 脚手架**是 `hc-tech-design` / `hc-dev` 的活**，接入 skill **不越界**——建了 `projects/<名>/` + `AGENTS.md` + `CLAUDE.md` 就停，别顺手 `mkdir src/ internal/ cmd/`、别生成脚手架代码。

### 第 3 步 · 记第一个决策 ADR
- **问什么**：把第 1–2 步定下的**选型 / 结构为什么这么定**跟用户对齐——哪些是用户拍的、备选是什么、为什么排除。
- **确认什么**：ADR 的决策 / 备选 / 理由跟用户核对，别替用户编理由。
- **落什么**：一条**项目自己的决策记录 ADR**（落项目的决策记录处，具体路径由本步与用户约定 / 沿用项目约定），写清：**背景 → 决策 → 备选 + 为什么排除 → 影响**。这是这个项目"第一个决策留痕"，往后决策接着记。

### 第 4 步 · 接执行口（接线、不接假内容）
把项目的执行口**接上线**（不接假内容、不静默空）：

- **问什么**：这项目将来怎么验证——verify（项目自检）/ unit（单测）/ api（接口测）/ e2e（端到端）/ **sandbox 三字段**（`sandbox` 起 / `sandbox_down` 停 / `sandbox_status` 查，契约见 `SANDBOX_CONTRACT.md`；可选 `sandbox_reset` / `sandbox_seed`）**这几条接入点，每条现在能接实吗**？将来脚本放哪、叫什么命令？
- **确认什么**：每条接入点的**三态取值**（见 ⑤）跟用户逐条确认——现在有真命令就填真命令；接不了就填 `PENDING:` 并写清楚为什么现在空 / 补的条件；这项目压根不需要就填 `N/A:` 并写理由。
- **落什么**：
  - 在 `workspace/verification.yaml` 给项目**占一条**（verify / unit / api / e2e / sandbox·sandbox_down·sandbox_status 各按三态取值）——**注意隔离纪律**：本 skill **只教用户怎么标、教 reviewer 怎么审**，实际写 `workspace/verification.yaml`、接 CI、建机检脚本是**主 agent 串行接线**做（本 skill 不代跑 index/regen、不代改 workspace / CI）；
  - 约定 **CI 认得它**（口子留好）；
  - 约定**将来脚本放哪、叫什么命令**（命令名先约定，**脚本本体先占位**）。
- **sandbox 特别说明**：sandbox 复杂、**由 `hc-create-sandbox` skill 单独接实**（契约见 `docs/harness/SANDBOX_CONTRACT.md`，ADR-0019）；本 skill 里 sandbox **只留占位**（三态里多半是 `PENDING:` 或 `N/A:`），不在这里搭沙箱。**verify / ci 是本 skill 内部小步骤**、不单独成 skill。

### 第 5 步 · 对抗评审搭出来的骨架
- **派 `hc-onboard-reviewer`** 子 agent 对抗挑刺搭出来的骨架（骨架完整性 / 三态占位是否合规 / 隔离边界有没有踩 / 是否越界搭了代码结构），**回改到过**（详见 ⑥）。

### 第 6 步 · 收尾
- **`make verify` 绿**：控制面自检通过（含结构 / 文档 / 索引）。
- **项目登记好**：项目在控制面被认得（登记 / 索引由主 agent 串行跑，本 skill 不代跑）。**"登记好"的完整口径 = 「引入关联进主目录」五项，新老项目共用**（被管工程登记 / CLAUDE.md shim / 根级索引 / CI / 加载链通，清单见 ④ 第 5 步）——新项目在第 2 · 4 步已分头覆盖大半，收尾按那五项对一遍、缺哪项补哪项。
- **确认什么**：把"verify 绿 + 登记好"的**真实证据**摆给用户，没真跑通不许声称通过（rule-0002 / rule-0003）。

### 第 7 步 · 交棒
- 骨架**过审 + verify 绿**后，**指路下游**：往后开发走 `hc-prd`（产需求）→ `hc-tech-design`（研发方案 / 接口设计）→ `hc-dev`（写代码）。
- **第一个需求包按 rule-0001 立**：接入完成不等于可以直接写业务码——**用户可见的需求 / 行为 / 验收目标**要先在 `docs/features/` 立需求包并就绪，未就绪 MUST STOP。交棒时把这条讲给用户。

## ④ 老项目 8 步（每步：问什么 / 确认什么 / 落什么）
老项目 = **有代码有历史**，取向是**倒着对齐**：把项目原有规范 / 老资产收进 harness 规矩，让"控制面记的 ↔ 项目真实代码"对得上、不漂。依据 **ADR-0018**。

> **铁律（老项目版，比新项目更吃重）**：agent 扫出来 / 理解的**不一定对**——每条先出"**我看到的**"草稿，用户**逐条 √/✗/改**，**确认过的才落**；**绝不"扫到就当规范落"**（rule-0008）。
>
> **红线（贯穿老分支）**：**只收规范 / 记录、不改业务代码**——老分支产的是文档 / 规范 / 记录（AGENTS.md / rules / ADR / verification 条目），**不动项目业务代码**（改码 / 重构归 `hc-dev`，越界）。

### 第 1 步 · 定位接管
- **问什么**：确认这确实是"**已有代码 + 历史**"的老项目；代码放（或将放）`projects/<名>/`；接管**范围**是全仓还是某几块。
- **确认什么**：项目名 / 路径 / 范围跟用户核准；项目名与现有工程**不撞**（核 `projects/` 现有目录 + `workspace/verification.yaml` 现有 `name`，多工程纪律同新项目第 1 步）。
- **落什么**：先不落文件，确认的范围是后续所有步骤的边界。

### 第 2 步 · 拆模块（前置 + 引导对话）
- **问什么**：agent 先**通读项目整体结构**（目录 / 包 / 服务 / 分层）试着划模块——**边界拿不准 / 看不明白的当场问用户、引导一起划**（"X 和 Y 算一块还是两块？""目录 Z 干啥的、归哪块？"），**不静默假设、不脑补边界**（rule-0008）。
- **确认什么**：模块地图**全划清 + 用户确认**才作数——没有大块目录 / 模块漏在地图外没人扫。
- **落什么**：确认过的**模块地图**——它是第 3 步"按模块滚"的清单，**地图没确认不开滚**。

### 第 3 步 · 按模块滚（每模块：扫 → 确认 → 搬）
对确认过的每个模块**逐个**走三小步——**一模块一模块滚，别一口气啃完**（小项目模块少可一轮过）：
- **① 扫 + 理资产**：扫该模块的现有规范 / 技术栈 / 历史资产 / 该模块测试，出"**我看到的**"草稿——**只是草稿、不落**。
- **② 用户逐条确认**：草稿逐条 √/✗/改——✗ 的丢弃、改的按用户版本。
- **③ 搬进规范**：**只落确认过的**——该模块目录的 `AGENTS.md` + `CLAUDE.md` shim；大块内容独立成文落 docs；全项目通用的约束在项目根上补 rules（走 `hc-add-rule`）、历史决策补 ADR 记下来。

### 第 4 步 · 接执行口（发现 + 对齐，非占位起步）
老项目多半"**已有一套**"验证——本步是**发现 + 对齐现有的**，不是从空占位起步：
- **问什么**：扫出项目现有 verify / build / 测试 / CI / sandbox 配置，出"我看到的"清单。
- **确认什么**：清单跟用户逐条确认；CI 跟用户定「接进 harness affected-verify（**推荐**，口径统一）还是保留项目那套」。
- **落什么**：确认后接进来——verify / 单测 / api / e2e 现有命令**确认后填真命令**进 `workspace/verification.yaml`（仍守三态：有则真、无则 `PENDING:` / `N/A:`，见 ⑤）；sandbox 有现成就接真命令、没有填 `PENDING:` + 以后走 `hc-create-sandbox` 补。**接线纪律同新项目第 4 步**：skill 只教怎么标 / 怎么审，实际写 verification.yaml / 接 CI 由主 agent 串行做。

### 第 5 步 · 引入关联进主目录（最后一公里，五项，新老共用）
项目内落了东西**不等于**挂进了治理网——五项检查逐项过（新项目收尾也按这五项对，见 ③ 第 6 步）：
1. **登记被管工程**：CURRENT_STATUS 被管工程表（必要时根 AGENTS.md）；
2. **每个 AGENTS.md 配 CLAUDE.md shim**（就近发现不断链）；
3. **根级索引收录（只有 rules 半边是自动的）**：项目落的 rules（`AGENTS.md` 带 `<!-- rule -->` 标记）进 `docs/rules/index.yaml`（`rules-index.sh` 全仓扫、由主 agent 串行跑）；**项目 ADR 不进控制面 `docs/decisions/`**（手维护的控制面账本，项目内容进去违 rule-0015）——项目 ADR 落**项目自己的决策记录处**、项目内有可寻的登记 / 目录约定即可；
4. **CI 接根 `.github/workflows` affected-verify**（若第 4 步定的是接 harness 口径；**尚未建则建起**——现根 workflows 只有 `verify.yml`，affected-verify 首次接入时建，见 `docs/harness/CI.md`）；
5. **核加载链通**：根启动顺序 → 能就近加载到 `projects/<名>/AGENTS.md`。

### 第 6 步 · 对抗评审
- **派 `hc-onboard-reviewer`** 对抗挑刺，**老项目加 4 条判据**（忠于确认 / 对齐不漂 / 模块地图确认且覆盖全 / 引入关联五项齐，详见 ⑥），**回改到过**。

### 第 7 步 · 收尾
- **`make verify` 绿** + **登记好**（五项齐），真实证据摆给用户、没真跑通不许声称通过（rule-0002 / rule-0003），同新项目第 6 步。

### 第 8 步 · 交棒
- 往后开发走 `hc-prd` → `hc-tech-design` → `hc-dev`；**动业务码前先在 `docs/features/` 立需求包**（rule-0001）——接入只收了规范 / 记录，**不含任何业务代码改动的许可**。

## ⑤ 占位三态规矩（关键：占位不许静默空）
第 4 步每个接入点（verify / unit / api / e2e / sandbox·sandbox_down·sandbox_status，+可选 sandbox_reset / sandbox_seed——sandbox 拆三字段见 `docs/harness/SANDBOX_CONTRACT.md`）的值**必是三态之一**——**目标：占位看得见 + 绕不过去，防"开发过了没补"**（同 `test-cases-audit` 的「无 · 理由」逃生口思路）：

1. **真命令**（已接实）——直接写将来能跑的命令。
2. **`PENDING: <为啥现在空 / 补的条件>`**（待接实）——例：`PENDING: 首个接口落地后补 api 冒烟`。**`make verify` 会 warn 提醒**（机检脚本主 agent 另建），同时**项目 `AGENTS.md` 里留一条"待补"记录**，让 agent 以后进来看得见、绕不过去。
3. **`N/A: <理由>`**（这项目不需要这项）——例：`N/A: 纯 CLI 工具、无对外接口`。

**静默空 / 留白 / 裸 `TODO` = 红**。skill 只**教用户这么标、教 reviewer 这么审**；机检脚本本体由主 agent 另建。

## ⑥ 对抗评审（派 hc-onboard-reviewer 到过）
搭 / 收完（新项目第 5 步 / 老项目第 6 步），**派 `hc-onboard-reviewer` 子 agent 对抗挑刺**，**回改到过**（同 `hc-tech-design` 单 reviewer 线性 loop）：
- reviewer **只评不改**（无 Write），出结构化清单（位置 / 严重度 / 问题 / 证据 / 怎么补）；
- 主 agent（接入向导）据清单**回改 → 复审 → 到零缺陷**。
- **怎么派**：**Claude Code** 用 workflow / Task（`agent(..., {agentType:'hc-onboard-reviewer'})` 循环，会话模型）；**Codex** 派**同名**双栈 reviewer。
- **reviewer 审什么**（总览，约束本体在 reviewer 子 agent 上下文，共 6 块）：① **骨架最小没越界**（只搭壳、没搭代码结构 / 脚手架；**老分支该块口径改判**——老工程本就有代码，越界=「本次接入改了项目业务代码」）② **AGENTS.md 红线合理 + 有 shim**（精简、该下沉的下沉、不复刻全局）③ **选型 ADR 有据**（决策 / 备选 / 理由齐、用户拍过、引用不悬空）④ **verification.yaml 三态合规**（每接入点是真命令 / `PENDING:` / `N/A:` 之一，无静默空 / 裸 TODO，PENDING 有 AGENTS.md 待补记录）⑤ **忠于用户确认的选择**（没替用户擅自定）⑥ **隔离没搞反**（项目产物写真实领域词是**对**的；反过来 skill / reviewer / 模板不掺项目内容，见 ⑦）。
- **老项目新增 4 条判据**（**只在老分支触发**，别拿去误伤新项目骨架；新项目原判据不动）：
  - a) **忠于确认**：落的每条规范 / 规则 / ADR 是不是用户确认过的（√）——把 agent 臆测 / 未确认项当规范落 = **major**（rule-0008）；
  - b) **对齐不漂**：控制面记的（AGENTS.md / rules / ADR / verification 条目）↔ 项目真实代码 / 目录**对得上**——指了不存在的目录、记了与代码矛盾的规范 = 漂移（能 grep / ls 核就核）；
  - c) **模块地图确认过且覆盖全**：地图经用户确认、没有大块目录 / 模块漏在地图外没人扫；
  - d) **引入关联全**：被管工程登记 / shim / 根级索引 / CI / 加载链**五项齐**——缺 = 项目没真挂进治理网。

## ⑦ 硬规则
- **只搭壳、不越界**（新项目）：建 `projects/<名>/` + 精简 AGENTS.md + CLAUDE.md shim 就停，**不搭代码结构 / 分层 / 脚手架 / 实现**（那是 `hc-tech-design` / `hc-dev` 的活）。
- **扫到的先确认再落**（老项目命根，rule-0008）：agent 扫出来 / 理解的**不一定对**——每条先出"我看到的"草稿，用户**逐条 √/✗/改**，确认过的才落；**绝不"扫到就当规范落"**。
- **只收规范 / 记录、不改业务代码**（老项目）：老分支产的是文档 / 规范 / 记录（AGENTS.md / rules / ADR / verification 条目），**不动项目业务代码**——改码 / 重构归 `hc-dev`，越界。
- **模块地图确认才滚**（老项目）：地图**全划清 + 用户确认**才开始按模块滚；边界拿不准当场问用户、不静默假设、不脑补。
- **多工程隔离**：`projects/` 可能已有别的工程——接新工程**只动自己那份**（建自己的 `projects/<名>/`、给 `verification.yaml` **append** 自己那条、只落自己项目的 ADR / 规则），**绝不碰、不覆盖别的工程**的目录 / `AGENTS.md` / verification 条目 / 决策记录；项目名与路径**不与现有工程撞**（第 1 步先核）。
- **全程用户确认、不替用户定**：选型 / 结构 / 规矩 / 接入点取值都**摆选项 + 讲取舍让用户拍**，不"看见啥落啥"、不静默替用户拍板；不确定就查 + 问（rule-0008）。
- **占位不静默**：接入点守三态（真命令 / `PENDING: <条件>` / `N/A: <理由>`），**静默空 / 留白 / 裸 TODO = 红**；PENDING 同步在项目 AGENTS.md 留"待补"记录。
- **通用 / 项目隔离别搞反**（rule-0015）：**skill 本体 / hc-onboard-reviewer / 模板**必须**通用、不掺具体项目内容**（中性占位，不预设某语言 / 框架 / 多租户 `tenant_id`）；但本 skill **产出的项目骨架**（`projects/<名>/AGENTS.md`、第一个 ADR、`verification.yaml` 项目条目）**本就项目专属、写满真实领域词是对的**。方向别反：通用的归资产、具体的归产物。
- **接线归主 agent**：本 skill **不代改** `workspace/verification.yaml` / CI、**不代建**机检脚本、**不代跑** index / regen / ADR 定稿——skill 教用户怎么标、教 reviewer 怎么审，接线由主 agent 串行做。
- **收尾不假完成**：`make verify` 绿 + 登记好要有**真实证据**（rule-0002 / rule-0003）；交棒时提醒第一个需求包按 rule-0001 立。
- **对抗评审到过**：派 `hc-onboard-reviewer` 回改到零缺陷。

## ⑧ 演进（rule-0007）
接入流程 / 骨架结构 / 三态占位规矩 / 评审机制 / 分流（新 / 老两分支）变化时回顾本 skill（连同 `.claude/agents/hc-onboard-reviewer.md`、`.codex/agents/hc-onboard-reviewer.toml`、及将来的接入模板）；改完同步 `version` / `last_reviewed`，跑 `bash scripts/skills-index.sh`（本步由主 agent 串行做，skill 本身不代跑）。
