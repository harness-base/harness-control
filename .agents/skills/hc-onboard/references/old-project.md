# 老项目 8 步（详细版）

> `hc-onboard` 走**老项目**分支（②入口分流判为"已有代码 + 历史的老工程"）时读。按需读、不占常驻。
> 老项目 = **有代码有历史**，取向是**倒着对齐**：把项目原有规范 / 老资产收进 harness 规矩，让"控制面记的 ↔ 项目真实代码"对得上、不漂。依据 **ADR-0018**。
> 步骤编号与 SKILL.md 的 ④ 骨架一一对应（对抗评审=第 6 步、收尾=第 7 步、交棒=第 8 步）——`hc-onboard-reviewer` 按这套步号回指，别改号。

> **铁律（老项目版，比新项目更吃重）**：agent 扫出来 / 理解的**不一定对**——每条先出"**我看到的**"草稿，用户**逐条 √/✗/改**，**确认过的才落**；**绝不"扫到就当规范落"**（rule-0008）。
>
> **红线（贯穿老分支）**：**只收规范 / 记录、不改业务代码**——老分支产的是文档 / 规范 / 记录（AGENTS.md / rules / ADR / verification 条目），**不动项目业务代码**（改码 / 重构归 `hc-dev`，越界）。

## 第 1 步 · 定位接管
- **问什么**：确认这确实是"**已有代码 + 历史**"的老项目；代码放（或将放）`projects/<名>/`；接管**范围**是全仓还是某几块。
- **确认什么**：项目名 / 路径 / 范围跟用户核准；项目名与现有工程**不撞**（核 `projects/` 现有目录 + `workspace/verification.yaml` 现有 `name`，多工程纪律同新项目第 1 步）。
- **落什么**：先不落文件，确认的范围是后续所有步骤的边界。

## 第 2 步 · 拆模块（前置 + 引导对话）
- **问什么**：agent 先**通读项目整体结构**（目录 / 包 / 服务 / 分层）试着划模块——**边界拿不准 / 看不明白的当场问用户、引导一起划**（"X 和 Y 算一块还是两块？""目录 Z 干啥的、归哪块？"），**不静默假设、不脑补边界**（rule-0008）。
- **确认什么**：模块地图**全划清 + 用户确认**才作数——没有大块目录 / 模块漏在地图外没人扫。
- **落什么**：确认过的**模块地图**——它是第 3 步"按模块滚"的清单，**地图没确认不开滚**。

## 第 3 步 · 按模块滚（每模块：扫 → 确认 → 搬）
对确认过的每个模块**逐个**走三小步——**一模块一模块滚，别一口气啃完**（小项目模块少可一轮过）：
- **① 扫 + 理资产**：扫该模块的现有规范 / 技术栈 / 历史资产 / 该模块测试，出"**我看到的**"草稿——**只是草稿、不落**。
- **② 用户逐条确认**：草稿逐条 √/✗/改——✗ 的丢弃、改的按用户版本。
- **③ 搬进规范**：**只落确认过的**——该模块目录的 `AGENTS.md` + `CLAUDE.md` shim；大块内容独立成文落 docs；全项目通用的约束在项目根上补 rules（走 `hc-add-rule`）、历史决策补 ADR 记下来。

## 第 4 步 · 接执行口（发现 + 对齐，非占位起步）
老项目多半"**已有一套**"验证——本步是**发现 + 对齐现有的**，不是从空占位起步：
- **问什么**：扫出项目现有 verify / build / 测试 / CI / sandbox 配置、**接口清单导出方式**（routelist：有现成命令 / proto / OpenAPI 就着它，确认后落导出脚本），出"我看到的"清单。
- **确认什么**：清单跟用户逐条确认；CI 跟用户定「接进 harness affected-verify（**推荐**，口径统一）还是保留项目那套」。
- **落什么**：确认后接进来——verify / 单测 / api / e2e / routelist 现有命令**确认后填真命令**进 `workspace/verification.yaml`（仍守三态：有则真、无则 `PENDING:` / `N/A:`，见 SKILL.md ⑤）；routelist **输出须一行一个接口**——现成命令不满足就包一层导出脚本；sandbox 有现成就接真命令、没有填 `PENDING:` + 以后走 `hc-create-sandbox` 补。**接线纪律同新项目第 4 步**：skill 只教怎么标 / 怎么审，实际写 verification.yaml / 接 CI 由主 agent 串行做。

## 第 5 步 · 引入关联进主目录（最后一公里，五项，新老共用）
项目内落了东西**不等于**挂进了治理网——五项检查逐项过（新项目收尾也按这五项对，见 `references/new-project.md` 第 6 步）：
1. **登记被管工程**：CURRENT_STATUS 被管工程表（必要时根 AGENTS.md）；
2. **每个 AGENTS.md 配 CLAUDE.md shim**（就近发现不断链）；
3. **根级索引收录（只有 rules 半边是自动的）**：项目落的 rules（`AGENTS.md` 带 `<!-- rule -->` 标记）进 `docs/rules/index.yaml`（`rules-index.sh` 全仓扫、由主 agent 串行跑）；**项目 ADR 不进控制面 `docs/decisions/`**（手维护的控制面账本，项目内容进去违 rule-0015）——项目 ADR 落**项目自己的决策记录处**、项目内有可寻的登记 / 目录约定即可；
4. **CI 接根 `.github/workflows` affected-verify**（若第 4 步定的是接 harness 口径；**尚未建则建起**——现根 workflows 只有 `verify.yml`，affected-verify 首次接入时建，见 `docs/harness/CI.md`）；
5. **核加载链通**：根启动顺序 → 能就近加载到 `projects/<名>/AGENTS.md`。

## 第 6 步 · 对抗评审
- **派 `hc-onboard-reviewer`** 对抗挑刺，**老项目加 4 条判据**（忠于确认 / 对齐不漂 / 模块地图确认且覆盖全 / 引入关联五项齐，详见 SKILL.md ⑥），**回改到过**。

## 第 7 步 · 收尾
- **`make verify` 绿** + **登记好**（五项齐），真实证据摆给用户、没真跑通不许声称通过（rule-0002 / rule-0003），同新项目第 6 步。

## 第 8 步 · 交棒
- 往后开发走 `hc-prd` → `hc-tech-design` → `hc-dev`；**提示第一份需求建议先走 `hc-prd`**（提示、非门禁，ADR-0023）——接入只收了规范 / 记录，改业务代码是下游 `hc-dev` 的事、不在本次接入范围。
