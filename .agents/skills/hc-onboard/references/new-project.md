# 新项目 7 步（详细版）

> `hc-onboard` 走**新项目**分支（②入口分流判为"从零起的新项目"）时读。按需读、不占常驻。
> 每步：**问什么 / 确认什么 / 落什么**。一条连贯的引导对话，主 agent 当接入向导，**每步先确认再落**；每步都守铁律（摆选项 + 讲取舍，用户拍板）。依据 **ADR-0017**。
> 步骤编号与 SKILL.md 的 ③ 骨架一一对应（对抗评审=第 5 步、收尾=第 6 步、交棒=第 7 步）——`hc-onboard-reviewer` 按这套步号回指，别改号。

## 第 1 步 · 收基本信息
- **问什么**：项目名（**kebab-case**，将来是 `projects/<名>/` 目录名）/ 目标（这工程要解决什么）/ 概述（一句话它是什么）/ 技术栈（框架 · 语言）。**怎么问由 agent 临场把握**——用户一次给全就不必逐条追；给得零碎 / 含糊就补问，别拿"已有信息 + 想要的结果"硬编（rule-0008）。
- **确认什么**：把收齐的四项**复述给用户确认**——尤其项目名是不是想要的 kebab、栈选型用户是否认可（栈有取舍就摆出来让用户拍，别默认替他选）。**项目名必须跟现有工程不撞**：`projects/` 是**多工程**目录（可能已挂别的工程），动手前先核 `projects/` 现有目录名 + `workspace/verification.yaml` 现有 `name`，撞了让用户另起一个唯一的 kebab 名，绝不覆盖 / 混进现有工程。
- **落什么**：**先不落文件**，把确认过的信息作为后续步骤的输入。

## 第 2 步 · 搭最小骨架（只搭壳、不搭代码结构）
- **问什么**：这一步主要是"搭"不是"问"，但**红线 / 指针要不要项目专属项**得跟用户核（如这项目有没有额外红线）。
- **确认什么**：给用户看将要建的骨架清单，确认了再写。
- **落什么**：
  - `projects/<名>/` 目录；
  - 精简 **`AGENTS.md`**——只放**项目红线 / 技术栈 / 指针**（指向控制面根 `AGENTS.md` 与产物区），**不复刻**全局规则、不堆流水账；
  - 同级 **`CLAUDE.md` shim**（`@AGENTS.md`，让 Claude Code 就近发现）。
- **红线（本步命根）**：**只搭壳、不搭代码结构**。目录布局 / 分层 / 包结构 / 脚手架**是 `hc-tech-design` / `hc-dev` 的活**，接入 skill **不越界**——建了 `projects/<名>/` + `AGENTS.md` + `CLAUDE.md` 就停，别顺手 `mkdir src/ internal/ cmd/`、别生成脚手架代码。

## 第 3 步 · 记第一个决策 ADR
- **问什么**：把第 1–2 步定下的**选型 / 结构为什么这么定**跟用户对齐——哪些是用户拍的、备选是什么、为什么排除。
- **确认什么**：ADR 的决策 / 备选 / 理由跟用户核对，别替用户编理由。
- **落什么**：一条**项目自己的决策记录 ADR**（落项目的决策记录处，具体路径由本步与用户约定 / 沿用项目约定），写清：**背景 → 决策 → 备选 + 为什么排除 → 影响**。这是这个项目"第一个决策留痕"，往后决策接着记。

## 第 4 步 · 接执行口（接线、不接假内容）
把项目的执行口**接上线**（不接假内容、不静默空）：

- **问什么**：这项目将来怎么验证——verify（项目自检）/ unit（单测）/ api（接口测）/ e2e（端到端）/ **sandbox 三字段**（`sandbox` 起 / `sandbox_down` 停 / `sandbox_status` 查，契约见 `SANDBOX_CONTRACT.md`；可选 `sandbox_reset` / `sandbox_seed`）/ **routelist**（导出实现侧接口清单的命令，输出一行一个接口——契约对照场景消费，ADR-0026；接口定义在哪[proto / OpenAPI / 路由注册]、怎么导最稳，跟用户聊定）**这几条接入点，每条现在能接实吗**？将来脚本放哪、叫什么命令？
- **确认什么**：每条接入点的**三态取值**（见 SKILL.md ⑤）跟用户逐条确认——现在有真命令就填真命令；接不了就填 `PENDING:` 并写清楚为什么现在空 / 补的条件；这项目压根不需要就填 `N/A:` 并写理由。
- **落什么**：
  - 在 `workspace/verification.yaml` 给项目**占一条**（verify / unit / api / e2e / sandbox·sandbox_down·sandbox_status / routelist 各按三态取值）——**注意隔离纪律**：本 skill **只教用户怎么标、教 reviewer 怎么审**，实际写 `workspace/verification.yaml`、接 CI、建机检脚本是**主 agent 串行接线**做（本 skill 不代跑 index/regen、不代改 workspace / CI）；
  - 约定 **CI 认得它**（口子留好）；
  - 约定**将来脚本放哪、叫什么命令**（命令名先约定，**脚本本体先占位**）。
- **sandbox 特别说明**：sandbox 复杂、**由 `hc-create-sandbox` skill 单独接实**（契约见 `docs/harness/SANDBOX_CONTRACT.md`，ADR-0019）；本 skill 里 sandbox **只留占位**（三态里多半是 `PENDING:` 或 `N/A:`），不在这里搭沙箱。**verify / ci 是本 skill 内部小步骤**、不单独成 skill。

## 第 5 步 · 对抗评审搭出来的骨架
- **派 `hc-onboard-reviewer`** 子 agent 对抗挑刺搭出来的骨架（骨架完整性 / 三态占位是否合规 / 隔离边界有没有踩 / 是否越界搭了代码结构），**回改到过**（详见 SKILL.md ⑥）。

## 第 6 步 · 收尾
- **`make verify` 绿**：控制面自检通过（含结构 / 文档 / 索引）。
- **项目登记好**：项目在控制面被认得（登记 / 索引由主 agent 串行跑，本 skill 不代跑）。**"登记好"的完整口径 = 「引入关联进主目录」五项，新老项目共用**（被管工程登记 / CLAUDE.md shim / 根级索引 / CI / 加载链通，清单见 `references/old-project.md` 第 5 步）——新项目在第 2 · 4 步已分头覆盖大半，收尾按那五项对一遍、缺哪项补哪项。
- **确认什么**：把"verify 绿 + 登记好"的**真实证据**摆给用户，没真跑通不许声称通过（rule-0002 / rule-0003）。

## 第 7 步 · 交棒
- 骨架**过审 + verify 绿**后，**指路下游**：往后开发走 `hc-prd`（产需求）→ `hc-tech-design`（研发方案 / 接口设计）→ `hc-dev`（写代码）。
- **提示第一份需求建议先走 `hc-prd`**（提示、非门禁——skill 间松耦合、各 skill 可独立工作，ADR-0023）：需求理清（用户故事+AC 经确认）再进设计 / 实现，返工最少；但用户口述清楚直接走 `hc-dev` 也行。交棒时把这条讲给用户。
