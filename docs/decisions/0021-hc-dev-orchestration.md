# ADR-0021：hc-dev 编排式重构

> **更新（2026-07-08）**：吃上游第 ① 条"需求包（`docs/features/`，rule-0001 门禁不变）"与"保留不动"里的"需求包门禁"已随 [ADR-0023](0023-features-retirement.md) 改为"需求产出 `docs/prds/`（有则吃、无则提示走 `hc-prd`、可独立干）"——门禁不再保留。下方叙述保留历史原样。

## 背景

`hc-dev`（ADR-0009，6-28）建得早、没跟上这波：① 上游 `hc-tech-design` 的 `design.md` + `api-contract.md` 已存在，hc-dev 的 plan 还是"从头想"、只有一句回指；② 实现↔契约无人对账；③ UI/UX 还原靠"读 HTML 觉得像"，不准（用户点名）；④ 大活单 agent 串行写，没利用"契约定死后各层可并行"的架构优势（用户定：dev 一定是多 subagent）；⑤ 交棒口径旧。coding 是用户点名的**重中之重**。

## 决策

1. **hc-dev = 开发总监（编排式，形态 ≈ hc-test）**：主 agent 当总监——吃上游 → 按**改动面**定编制 → 派 worker 写码（契约为接缝）→ 对抗 review → 收尾交棒。**纪律全程常开、编制按活伸缩**。

2. **吃上游（输入优先级，有则必吃）**：① 需求包（`docs/features/`，rule-0001 门禁不变）② **设计方案 `docs/designs/<id>/design.md` + `api-contract.md`**——**有方案就照方案拆任务、不重新设计；方案定死的接口 / 数据模型写码时不许擅改**；③ PRD / 用户口述（无方案的小活）。**写码中发现方案有问题 → 停，回 `hc-tech-design` 改方案（用户参与），改完再继续——不许绕过契约自行其是**（防实现与设计漂移）。涉及对外接口 / 数据模型设计而**无契约** → 先走 `hc-tech-design`（原回指保留、升为硬门）。

3. **多 worker 编制（用户定），源驱动分层**：
   - **拆分依据 = 被管工程的真实分层**（前端 / 后端 / 客户端是常见例，**不硬编 worker 种类**——按项目结构 / 模块地图 / 设计方案临场定，项目没有的层不凑，rule-0015 同款源驱动）。
   - **按改动面伸缩**：单层小活 / 改 bug——总监直做或派 1 个 worker（不为琐事 spawn 编队，既有工作方式原则）；**跨层大活才按层并行**。
   - **契约为接缝**：并行的各层 worker 都以 `api-contract.md` 为准——契约已定死，前后端可独立并行、互不等待（这正是设计阶段单独出契约的意义）；契约没覆盖的跨层接缝 → 回 hc-tech-design 补契约再并行。
   - **并行纪律**：worker 界限按层 / 目录 **disjoint**（天然不重叠，防踩踏）；worker 继承全程纪律（不假设 / 决策点上报总监问用户 / 防技术债 / 验证如实）。
   - **新建 `hc-dev-worker` 双栈子 agent**（通用写码 worker）：领"一层 / 一个范围 + 契约 + 纪律"派单干活——**一种 worker 类型、多实例按层派**，不为每层造专职 agent。

4. **实现 ↔ 契约对账（新增环）**：写完后对照 `api-contract.md` 逐端点 / 字段 / 错误码核实现没漂——落在 `hc-code-reviewer` 新增判据（有契约时硬核，无契约跳过），与 api 用例线的"用例↔契约对账"呼应，三方（设计 / 实现 / 测试）都锚同一份契约。

5. **UI/UX 还原纪律（用户点名）**：涉视觉还原时——① **边写边看渲染结果**（起本地预览 / 浏览器，不写完脑补）；② **校验用视觉两件套**：渲染截图 ↔ 设计稿 / 原型对布局与观感 + 浏览器 inspect 读**计算后样式**核精确值（颜色 / 字号 / 间距）；③ **状态过全**（四态 + hover / 禁用，原型可点的照点）；④ **"读源码觉得对" = 不算验证**——无渲染证据不许声称"还原了"（rule-0009：视觉产物的真实信号是像素不是源码）。工具按平台现实（Claude Code preview / 浏览器工具；无渲染工具则请用户跑起来供截图）。写进 skill 纪律 + `hc-code-reviewer` 判据。

6. **保留不动**：两级纪律精神（升级信号并入"改动面定编制"）、改 bug 子模式（复现→真根因→先红测试→修→防回归）、重构 / 迁移子模式、需求包门禁、TDD 优先、superpowers 借用、验证如实、写完提醒用户测。**交棒口径更新**：写完 → 提醒用户测 + 指路 `hc-test`（用例线已实现；脚本线 / 回归占位，就绪后自动衔接）。

## 受影响（rule-0007）

- `hc-dev` SKILL 重构（总监形态，version bump）；ADR-0009 加前向指针（形态由本 ADR 演进，历史叙述不改写）。
- 新增 `hc-dev-worker` 双栈（注册 `.codex/config.toml`）。
- `hc-code-reviewer` 双栈更新：新增「实现↔契约对账」「UI 视觉还原证据」两块判据（version 语义不变，判据扩）。
- 上下游互指核措辞：`hc-tech-design`（下游按契约实现）/ `hc-onboard`（交棒链）/ `hc-test`（上游开发完成）——多为已一致，逐个核。
- 索引 regen。

## 备选（拒）

- **按功能模块拆 worker（而非按层）**：拒——功能贯穿多层，worker 文件面重叠、踩踏风险高；按层拆目录天然 disjoint、技能聚焦、契约作接缝。
- **硬编 前端/后端/客户端 三种专职 worker agent**：拒——预设项目形态违源驱动（有的项目没前端 / 多服务 / 有小程序端）；一种通用 `hc-dev-worker` 多实例按层派。
- **每层配专职 reviewer**：拒（暂）——通用 `hc-code-reviewer` 多实例多视角已支持，拆层 reviewer 维护翻倍；用起来不够再说。
- **UI 还原靠读源码 + 单元测试**：拒——用户点名不准；视觉产物的真实信号是渲染结果。

## 影响 / follow-up

- 编码环跟上全链路：需求（hc-prd）→ 设计（hc-tech-design）→ **开发（hc-dev 编排）** → 测试（hc-test）；三方锚同一契约。
- follow-up：真项目大活实战检验多 worker 并行（等全流程建完统一那批）；脚本线 / 回归就绪后 hc-dev 交棒口径再接一环。
