# ADR-0018：hc-onboard 老项目分支

## 背景

ADR-0017 建了 `hc-onboard` 的**新项目**分支（从零照规范搭骨架、顺着长），老项目分支留占位。本 ADR 填它。

**老项目 = 有代码、有历史**：不是从零生成，而是**倒着对齐**——把项目原有的规范 / 老资产**收进 harness 规矩**，让"控制面记的 ↔ 项目真实代码"对得上、不漂。与新项目正好反方向。

## 决策

1. **同一个 `hc-onboard` skill 的第二条岔路**（不是新 skill）：入口 ② 先问 新/老，本 ADR 落"老"这条。接执行口 / 评审 / 收尾 / 交棒的骨架与新项目共用，但老项目有自己的取向（发现 + 对齐现有的，而非占位）。

2. **全程用户确认（铁律，比新项目更吃重）**：agent 扫出来、理解的**不一定对**——每条先出"**我看到的**"清单草稿，用户**逐条 √ / ✗ / 改**，确认过的才落。绝不"扫到就当规范落"（rule-0008：外部 / 现状材料先验收）。

3. **老项目流程（8 步；核心是"先拆模块 → 按模块滚 → 引入关联进主目录"）**：
   1. **定位接管**：确认是"已有代码 + 历史"的老项目、放 `projects/<名>/`、拿到范围。
   2. **拆模块（前置——不先拆，无从按模块整理；且是引导对话、不是 agent 自己拍一版）**：agent 通读项目整体结构（目录 / 包 / 服务 / 分层）试着划模块，**边界拿不准 / 看不明白的当场问用户、引导一起划**（如"X 和 Y 算一块还是两块""目录 Z 干啥的、归哪块"），**不静默假设、不脑补边界**（rule-0008，防理解跑偏）；来回收敛到模块地图**全划清 + 用户确认**才作数。这份确认过的模块地图 = 后面"按模块滚"的清单。
   3. **按模块滚动（每个模块：扫 → 确认 → 搬）**：对确认过的每个模块逐个走——① **扫 + 理资产**（现有规范 / 技术栈 / 历史资产 / 该模块测试，出"我看到的"草稿、不直接落）→ ② **用户逐条确认**（√ / ✗ / 改）→ ③ **搬进规范**（只落确认过的：该目录写 `AGENTS.md` + `CLAUDE.md` shim、大块独立成文 docs、根上补 rules 走 `hc-add-rule`、补 ADR 把项目**历史决策**记成决策记录）。一模块一模块滚、别一口气啃完；小项目模块少可一轮过。
   4. **接执行口（老项目多半"已有一套"——发现 + 对齐，非占位）**：扫出项目现有 verify / build / 测试 / CI / sandbox 配置 → 用户确认 → 接进来：verify / 单测 / api / e2e 现有命令确认后填**真命令**进 `verification.yaml`（三态：有则真、无则 `PENDING:` / `N/A:`）；CI 跟用户定「接进 harness affected-verify（推荐、口径统一）还是保留项目那套」；sandbox 有现成就接真命令、没有 `PENDING:` + 以后走 `hc-create-sandbox` 补（ADR-0019）。
   5. **引入关联进主目录（接入的"最后一公里"，新老项目都要）**：把项目挂进控制面治理网——① 登记被管工程（`CURRENT_STATUS` 被管工程表、必要时根 `AGENTS.md`）；② 每个 `AGENTS.md` 配 `CLAUDE.md` shim（主目录加载链才认）；③ 根级索引收录（**只有 rules 半边是自动的**：项目 rules 经 `rules-index.sh` 全仓扫进 `docs/rules/index.yaml`；**项目 ADR 不进控制面 `docs/decisions/`**——那是手维护的控制面账本，项目内容进去违 rule-0015——项目 ADR 落项目自己的决策记录处、项目内有可寻登记即可）；④ CI 接进根 `.github/workflows` 的 affected-verify（步骤 4 定的口径；**尚未建则建起**——现根 workflows 只有 `verify.yml`，affected-verify 首次接入时建）；⑤ 核加载链通（根 `AGENTS.md` 启动顺序 → 能就近加载到 `projects/<名>/AGENTS.md`）。
   6. **对抗评审**：派 `hc-onboard-reviewer`（老项目加判据，见 5），回改到过。
   7. **收尾**：`make verify` 绿、登记好。
   8. **交棒**：往后开发走 `hc-prd`/`hc-tech-design`/`hc-dev`；动业务码前立需求包（rule-0001）。

4. **先拆模块、再按模块滚（顺序不能反）**：拆模块（步骤 2）是"按模块整理"的**前提**——没有确认过的模块地图，无从按模块下手。

5. **`hc-onboard-reviewer` 加老项目判据（双栈，仅老分支触发）**：
   - **忠于确认**：落的每条规范 / 规则 / ADR 是不是用户确认过的，把 agent 臆测 / 未确认项当规范落 = major（rule-0008）。
   - **对齐不漂**：控制面记的（AGENTS.md / rules / ADR / verification 条目）↔ 项目真实代码 / 目录对得上——指了不存在的目录、记了与代码矛盾的规范 = 漂移（能 grep / ls 核就核）。
   - **模块地图确认过且覆盖全**：地图经用户确认、没有大块目录 / 模块漏在地图外没人扫。
   - **引入关联全**：登记 / shim / 根级索引 / CI / 加载链五项齐——缺 = 项目没真挂进治理网。

6. **只把现有的收进规范、不改业务代码**：老项目分支产的是**文档 / 规范 / 记录**（AGENTS.md / rules / ADR / verification 条目），**不动项目业务代码**（改代码 / 重构是 `hc-dev` 的活，越界）。

## 受影响

- `hc-onboard` skill：② 分流的老项目分支 占位 → 实现（新增老项目 8 步小节；「引入关联进主目录」新老共用）；version bump。
- `hc-onboard-reviewer`（双栈）：加老项目 4 条判据（仅老分支触发，不误伤新项目骨架）。
- `docs/harness/PROJECT_ONBOARDING.md`：速查里"老项目分支后续补"改已实现。
- `templates/project-agents.md`：「指针」段决策记录指向随 M2 口径同步——项目 ADR 指**工程自己的决策记录处**（非控制面根 `docs/decisions/`），并分支限定（新第 3 步 / 老第 3 步补记）；修悬空的"PROJECT_ONBOARDING 第 2 步"引用（eval 011 挖出，随本批修）。
- ADR-0017：加前向指针（老分支已由本 ADR 实现）。

## 备选（拒）

- **老项目单独成一个 skill**：拒——和新项目是同一件事（接工程）的两条岔路，后段共用；拆两个 skill 重复维护。
- **扫到就落、不逐条确认**：拒——agent 对老项目的理解不一定对（README 过时、注释撒谎、隐含约定），不确认就落 = 把臆测当规范污染控制面（rule-0008 红线）。这是老项目分支的**命门**。
- **不先拆模块、直接全项目扫**：拒——没有确认过的模块地图，"按模块整理"无从下手，大项目一口气啃必失控。
- **连业务代码一起重构对齐**：拒——越界抢 `hc-dev`；老分支只收"规范 / 记录"侧，代码照旧、由后续开发按需动。

## 影响 / follow-up

- `hc-onboard` 覆盖"新 + 老"两类工程，接入能力补全；老项目"拆模块 → 扫→确认→搬 → 挂进治理网"有据可依、有 reviewer 兜。
- follow-up：真接一个老项目实战检验这条链；"扫资产"量大时是否拆多 worker 编排（现按连贯对话，真需要再说）。
