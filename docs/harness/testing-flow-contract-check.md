# 接口契约对照线（testing-flow 分线·ADR-0026）

> `testing-flow.md`（总纲）的分线文件——**做契约对照时读本文**，按需读、不占常驻。
> 干什么：开发完成后，把**实现里实际暴露的接口清单**和**设计契约承诺的接口**对账，抓两边不一致。
> 分工（用户拍）：**取数用脚本**（routelist 接入点，确定性、可重复）、**对比判断用 agent**（语义对齐，两边格式不同不做机械 diff）。总监调度、**不新建 worker**。

## 门槛（两查，查的都是登记、不猜）

1. **契约在不在**：`docs/designs/<id>/api-contract.md` 不存在 → **本场景跳过**（没有"说好的"那半本账，无从对照；留一句为什么，不硬凑——松耦合"缺则略"）。
2. **routelist 是不是真命令**：查 `workspace/verification.yaml` 该工程的 `routelist` 字段——
   - **真命令** → 往下走；
   - **`PENDING:`** → 停下报告用户，指路补导出脚本（onboard 接执行口的活；工程接口定义在 proto / OpenAPI / 路由注册里，写个导出脚本即可），不硬跑、不让 agent 临场读代码提取（取数必须脚本化，可重复可审计）；
   - **`N/A:`**（无对外接口）→ 本场景跳过留痕。

## 步骤（总监直做或派通用 agent，一次性对照任务）

1. **脚本取数**：跑 `routelist` 登记的命令，拿**实现侧全量接口清单**（输出约定：一行一个接口——方法 + 路径，或 rpc 名）。
2. **agent 对比**（语义对齐，不是文本 diff——契约是 markdown 端点表格、清单是纯文本行，格式不同；且要容忍写法差异如 `:id` vs `{id}`、gRPC 方法名 ↔ HTTP 路径映射）：
   - **契约 → 清单**（每需求对照的主方向）：本需求 `api-contract.md` 端点索引里的每个端点，在全量清单里找得到吗？——找不到 = **实现不足**。
   - **清单 → 契约**（反向抓野接口，不必每需求做、收官 / 回归时全量做一次）：清单里有、**所有**契约都没记的接口——= **私自加的或契约漏记**。
3. **偏差清单 + 归因分发**（agent 判断、总监分发）：

| 偏差 | 归因 | 分发 |
|---|---|---|
| 契约有、实现没有 | 实现不足 | 回 `hc-dev` 补实现 |
| 实现有、契约没有 | 契约不足（漏记）或实现越界 | 复核根因：该有 → `hc-tech-design` 补契约 + api 用例线补用例；不该有 → `hc-dev` 收回 |
| 都有但方法 / 路径 / 形态对不上 | 看哪边错 | 对应分发（契约错改契约、实现错改实现） |

4. **报告**：偏差清单（含"无偏差"的如实结论）+ 分发去向，留 `docs/test-cases/<id>/`（或需求收官记录）可追溯。

## 与相邻机制的分工

- **hc-dev 的"实现↔契约对账"**（code review 判据）：逐字段对**本次改动涉及的端点**——改码时的深对账。
- **本场景**：收官时对**全量端点清单**——存在性对账（接口在不在），不逐字段。
- 两者互补：前者防"改的地方漂"，本场景防"漏了没实现 / 偷加了没记账"。

## 两层防线

| 谁 | 查什么 | 判据 |
|---|---|---|
| **脚本**（routelist） | 取数确定性——实现侧清单以脚本输出为准，不靠 agent 临场扒代码 | ADR-0026 |
| **agent** | 判断——语义对齐、偏差归因、分发去向 | rule-0008（不臆造）+ rule-0009（结论锚脚本真实输出） |
