# api 用例线（testing-flow 分线·ADR-0016）

> `testing-flow.md`（总纲）的分线文件——**做 api 用例线时读本文**，按需读、不占常驻（拆分依据 ADR-0024，内容自总纲原文搬移、口径未变）。

## 接口来源（硬地板，无源即停——与 e2e「缺则略」不同）

| 优先级 | 接口来源 | 拿它干嘛 |
|---|---|---|
| ① 最高 | **`docs/designs/<id>/api-contract.md`**（`hc-tech-design` 产的接口契约，ADR-0015） | 端点索引=接口清单、每接口错误表=业务异常列举、字段表/Mock=逐字段断言与夹具 |
| ② | **用户指定接口来源**（proto / OpenAPI / 路由表 / 现有接口代码） | 契约缺位时的替代真相源 |
| ③ | **都没有 → MUST STOP** | `hc-api-qa` 交回总监提醒用户"没接口来源、无法确定测什么"，绝不凭空臆造接口（rule-0008） |

> **源驱动、不预设**：协议（gRPC / HTTP-REST / MQ-event）、鉴权/限流等横切、Mock、字段、错误码——全按接口来源里**实际有什么**覆盖，不硬编项目假设（rule-0015）；契约有 Mock 就用契约 Mock、没有不硬凑。
> 错误只覆盖契约列的**约定内**错误（业务码 / 校验 400·422 / 鉴权 401·403 / 约定服务态如 503 DB_UNAVAILABLE）；**未约定的未预期故障**（裸 500 / panic）不在契约、不为它编用例。按「约定 / 未约定」分，不按状态码段一刀切。

## `hc-api-qa`（写用例）

- 覆盖 = **与接口来源一一对应**：契约 = 一份「接口清单」（协议无关）+ 每接口列举的「业务异常」（错误响应表）。① 每个接口都有对应用例、全覆盖；② 单接口列的每个业务异常各覆盖一个 case。
- 声明段①「## 接口清单」誊录契约端点索引（`- EP-NN：…`）、声明段②「## 业务异常」誊录各接口错误表（`- EX-NN：EP-NN · …`）；用例 `covers:` 挂 EP/EX（成功/边界 covers EP、失败 covers EX）。
- **正常逐字段断言、锚契约**（字段类型/枚举/格式/约束 + 状态断言，rule-0009）；**失败测约定内业务异常码、受控**（不裸 5xx/panic）；边界 / 等价类不可略。
- 套模板 `templates/api-test-case.md`；产物落 `docs/test-cases/<id>/`；**只写用例、不跑**（rule-0014）。约束写进 `hc-api-qa` 子 agent 上下文，不只靠模板。

## `hc-api-reviewer`（审用例）—— 3 块 + 来源门槛复核

1. **覆盖率**：对照接口来源，漏没漏接口 / 业务异常码 / 字段、是不是只测 happy 2xx、边界 / 等价类齐不齐。
2. **源符合 + 对账（api 核心）**：用例 ↔ 接口来源**回原文逐字段 / 逐错误码对照**——不臆造契约外端点 / 字段 / 错误码（rule-0008）、字段类型 / 约束 / 错误码语义 / 协议对得上、一一对应双向闭合。
3. **用例质量**：等价类 / 真边界、预期锚契约字段（rule-0009）、数据源用契约 Mock、牵强空壳。
4. **来源门槛复核**：QA 若在无接口来源下硬产用例 → **blocker**（违 rule-0008）。
- **只评不改**；出清单 → 总监派 `hc-api-qa` 回改 → 复审到零。判据 = 考题 015 + rule-0014 + rule-0009 + rule-0008。约束写进 reviewer 上下文。

## 用例模板（`templates/api-test-case.md`）

三段承重：`## 接口清单`（EP-NN）/ `## 业务异常`（EX-NN）/ `## 用例`（每条 `covers:` EP/EX + 字段：接口 / 类型[正常·边界·异常 + 等价类] / 请求参数[标等价类·边界值] / 预期[成功逐字段锚契约 · 失败业务异常码受控] / 数据来源[契约 Mock 或指定源]）。

## 两层覆盖防线

| 谁 | 查什么 | 判据 |
|---|---|---|
| **机器**（`test-cases-audit`） | **结构**：EP/EX ↔ 用例 `covers:` **双向闭合**（每个接口 / 业务异常都被覆盖 = 一一对应、covers 不引用未声明的 = 不臆造）+ 登记一致 | rule-0014 |
| **`hc-api-reviewer`** | **判断**：声明段 ↔ 契约原文对账（誊全没、字段准不准、有没有臆造）、覆盖够不够、质量 | 考题 015 + rule-0009 + rule-0008 |

> 两层「闭合」分清：机检查「声明段 ↔ 用例」文内自洽；reviewer 查「声明段 ↔ 契约原文」——契约里有、但声明段压根没誊进来的接口 / 错误码，机检看不见（它只在文内自洽），必须 reviewer 抓。
