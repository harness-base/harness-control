# 测试脚本线（testing-flow 分线·ADR-0024）

> `testing-flow.md`（总纲）的分线文件——**做测试脚本时读本文**，按需读、不占常驻。
> 触发：**开发彻底结束**（实现已合、用例已产）。与 rule-0014 的边界：**用例**（markdown 文档）只写不跑归 rule-0014；**脚本**（可执行代码）**写跑一体**——两种产物、两个口径，不冲突。

## 三步（写 → 跑+修 → 交付入池；回归是独立场景不在本线）

1. **写脚本**——把 `docs/test-cases/<id>/` 的用例翻译成可执行脚本：
   - **case 镜像用例**：一条用例一个 case，**case 名（函数名 / 测试名 / 文件名）锚用例编号 `TC-NN` + 线别消歧**——e2e 与 api 用例文档的编号**都是 `TC-n`、两个编号空间独立重叠**，脚本必须带线别区分（按线分目录 `test/<需求id>/{e2e,api}/`，或 case 名带线别前缀如 `e2e_tc01` / `api_tc01`）；「线别 + TC 号」合起来才是"一一对照"的可 grep 硬锚，reviewer 与人都能双向查；断言照用例的"预期"栏写（成功逐字段 / 失败异常码，锚唯一真实信号，rule-0009）。
   - **抽共享基础动作层**：api 侧（请求构造 / 鉴权 / 断言 helper）、e2e 侧（开页 / 点击 / 填表 / 等待），case 只写业务动作序列——helper 跨需求共享（如 `test/lib/`，位置随工程惯例），动作变了改一处、全部脚本跟着好。
   - **技术形态源驱动**（rule-0015）：语言 / 框架按工程真实栈定（bash / go test / playwright / pytest…），不硬编；看工程已有脚本先例（如既有 `test/` 下的形态）就跟。
2. **跑 + 修（sandbox 内调通）**——
   - **跑前卡门**：`sandbox`（up）→ `sandbox_status`（exit 0 才跑）→ 跑 → `sandbox_down`（见 `SANDBOX_CONTRACT.md`「运行时卡门」；命令取自 `workspace/verification.yaml`）。sandbox 未接实（PENDING）→ 停下报告用户，指路 `hc-create-sandbox`，不硬跑、不 mock 混过。
   - 跑**本需求**的 case（不带过往全量）；失败分两类处理：**脚本的问题改脚本**；**实现的问题不改脚本**——报出去回 `hc-dev` 修（脚本挖到真 bug 是它的功劳，别为绿而弱化断言）。
   - **完成定义**：本需求 case **全绿**，或**如实报告**残留失败清单 + 逐条归因（脚本 bug / 实现 bug / 环境缺失）——没跑过的测试脚本大概率是坏的，交付"没验证过的验证工具"= 假完成（rule-0002 / 0003）。
3. **交付入池**——脚本不是一次性验证，是**可复用回归资产**：
   - **落工程内** `projects/<工程>/test/<需求id>/`，**每需求独立目录**（不和过往混）；入口命令可挂 `workspace/verification.yaml` 的 `e2e` / `api` 接入点（按工程路由约定）。
   - **用例 ↔ 脚本双向登记**：用例文档（`docs/test-cases/<id>/`）记脚本目录路径；脚本 case 名锚用例编号——两边互查。
   - 存量池供**回归场景**（总纲第 6 步，独立、占位）统一跑。

## `hc-script-impl`（写 + 跑 + 修）

- 输入：`docs/test-cases/<id>/` 用例（**必需**——没有用例就停，交回总监提示先走用例线；不凭空造 case）、`api-contract.md`（api 脚本断言锚它，有则用）、工程真实栈与既有 test 形态（源驱动）。
- 干活按上面三步；**决策点上报总监**（如工程无既有脚本形态、框架二选一）——不静默替用户定（rule-0008）。
- 交付物：脚本文件（工程内）+ 运行报告（全绿 / 残留失败与归因）+ 用例文档的脚本路径登记。

## `hc-script-reviewer`（审）—— 两块（用户拍：对齐 + 明显 bug）

1. **对齐**：case ↔ 用例**一一对照**齐不齐——每条用例都有 case（漏 case）、每个 case 都锚真实用例（幽灵 case）、断言**忠于用例预期**（语义没测歪、没为绿弱化断言）。
2. **明显 bug**：断言空转 / 恒真（assert true、抓不住失败）、共享 helper 误用、**sandbox 卡门缺失**（不查 status 就跑）、硬编端口 / 环境 / 凭据、case 间共享可变状态（顺序耦合）、清理缺失（脏数据外溢）。
- **只评不改**；出结构化清单 → 总监派 `hc-script-impl` 回改 → 复审到零；**多视角并行对抗**（编排 pattern = `adversarial-review.md`，ADR-0022）。
- 运行报告如实性也在审（声称全绿要有运行输出证据，rule-0002 / 0003）。

## 两层防线

| 谁 | 查什么 | 判据 |
|---|---|---|
| **机器**（运行本身） | 脚本跑没跑通——运行输出即证据（全绿 / 失败清单） | rule-0002 / 0003 |
| **`hc-script-reviewer`** | **判断**：对齐（一一对照 / 忠于预期）+ 明显 bug（空转 / 卡门 / 硬编 / 耦合） | rule-0009 + ADR-0024 |

> 机检暂不建"case↔用例对照"的结构闸（脚本形态源驱动、各工程语言不同，静态解析不通用）；对齐靠 reviewer 判断层 + case 名锚点的可 grep 性兜。将来某工程形态稳定后可在工程内自建对照脚本。
