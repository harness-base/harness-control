# 统一回归线（testing-flow 分线·ADR-0027）

> `testing-flow.md`（总纲）的分线文件——**做统一回归时读本文**，按需读、不占常驻。
> 干什么：跑**过往需求攒下的存量脚本池**（脚本线交付物，`projects/<工程>/test/<需求id>/` 每需求一目录），确认**这次的改动没砸以前的功能**；不过就回改到过。
> 前置口径（总纲第 6 步）：**先本需求、再统一**——本需求的脚本在脚本线已跑绿（写跑一体），本线只管"统一"那半，**不一上来就跑带过往的全量**是指别拿全量回归替代本需求调通。

## 触发与门槛

- **触发**：脚本池有存量（工程 `test/` 下有 ≥1 个过往需求目录）+ 时机命中——大改收尾 / 新需求脚本入池后 / 用户点名"回归"。
- **门槛（sandbox 卡门，同脚本线）**：`sandbox`（up）→ `sandbox_status`（exit 0 才跑）→ 跑 → `sandbox_down`（命令取 `workspace/verification.yaml`；PENDING → 停下指路 `hc-create-sandbox`，不硬跑）。
- 池子为空（还没有任何需求脚本）→ 本场景无事可做，如实说、不硬造。

## 步骤（总监调度，跑 + 修脚本复用 `hc-script-impl`，不新建 worker）

1. **盘点池子**：列 `projects/<工程>/test/` 下全部需求目录（存量脚本池的清单就是目录本身，不另维护清单文件——目录即真相）。
2. **逐需求跑**：按需求目录挨个跑（顺序跑，防 case 间共享环境状态互相污染；各需求目录本就互相独立——脚本线的"每需求独立"纪律在此兑现价值）。
3. **汇总报告**：每需求 pass / fail 清单，**运行输出为证据**（rule-0002 / 0003——不许"应该都过"）；全绿 = 回归通过，如实报告收尾。
4. **失败归因两分**（agent 判断，总监分发）：

| 归因 | 什么样 | 处置 |
|---|---|---|
| **脚本过时** | 需求演进导致旧断言过期（接口改版、文案调整、流程变更——是**这次改动的预期效果**，旧脚本没跟上） | 派 `hc-script-impl` 更新旧脚本（锚新预期，别为绿弱化断言；对应用例文档若也过时，提醒走用例线同步） |
| **实现回归** | 这次改动**意外**破坏了以前的功能（旧行为本该保持） | 报 `hc-dev` 修实现（脚本不动——它抓到回归是它的功劳）；修完**重跑回归**到过 |

- 分不清是哪类 → 回到需求原始用例 / PRD 对照"旧行为该不该保持"，还分不清就**问用户**（rule-0008，不猜着改）。
5. **回改到过**：修复（无论哪类）后重跑受影响需求，全绿才算回归收尾；多轮不收敛 → 如实报告残留 + 归因，交用户裁。

## 两层防线

| 谁 | 查什么 | 判据 |
|---|---|---|
| **机器**（运行本身） | 池子跑没跑通——逐需求运行输出即证据 | rule-0002 / 0003 |
| **agent** | 判断——失败归因（脚本过时 vs 实现回归）、分发去向、"该不该保持"的裁定 | rule-0008（分不清就问）+ rule-0009（归因锚运行输出与用例原文） |

## 与脚本线的关系

- **脚本线**（`testing-flow-script.md`）：产出脚本 + 调绿**本需求**——它是池子的**生产者**。
- **本线**：消费池子、守**过往需求**不被新改动砸——它是池子的**消费者**。
- 同一批脚本资产、两个时机：入池时管自己绿，回归时管大家绿。
