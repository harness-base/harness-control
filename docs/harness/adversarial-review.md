# 对抗评审 pattern（多视角并行 review）

各 skill 的 **review 步的唯一真相源**（编排口径）。凡有"派 reviewer 对抗挑刺 → 回改到过"评审步的**产出型 skill**（如 `hc-dev` / `hc-tech-design` 等，**完整清单以 ADR-0022 受影响栏 + 各 skill review 步实际引用本文为准**，不在此复列）**引用本文、不复制细节**（摘要 + 指针的分层见下文「正式名」块）——改 review 编排只动这里。依据 **ADR-0022**。

> 立意：**review 不许单线程扫一遍就算**。单个 reviewer 一个视角容易漏；"单轮零 = 假收敛"（视角钝化，本仓实证：单轮挑干净、换怀疑视角又挖出 36+ 真问题）。所以 review 步统一走**多视角并行对抗 + 迭代收敛**。

## 核心 pattern（总监按此编排 review 步）

1. **fan out 多个 reviewer 实例、每个盯一个不同视角**（并行）——视角 = 该 review 域 reviewer 自带 rubric 的各块（如代码：correctness / 契约对账 / 安全 / 边界 / 缺测试；需求：覆盖 / 可观测 / 范围闭合；用例：覆盖 / 源符合 / 质量……**以各 reviewer 上下文的 rubric 为准，本文只示意、不复制全表**）。**同一个 reviewer 开 N 个同视角 = 浪费**，价值在视角相异。
2. **每条 finding 独立证伪**——默认怀疑"没问题"，主动找证据。
3. **汇总去重**——各视角 findings 合并、去重、按 blocker / major / minor 排。
4. **回改 → 复审**——总监据清单派对应 worker 回改，再 review。**每轮修复收尾做"声称清单 ↔ git status"对账**：修复若改了原清单之外的新文件，ADR 受影响栏 / 连带清单当轮回填；todo 勾选同口径——项里点名的每个文件都要在实改清单里，否则不许勾（lessons 同型三次：修复轮不回填 ×2、打勾没做 ×1）。
5. **迭代到一轮干净**——末轮那一遍"干净"必须**用没用过的新视角**再挑一遍（同视角单轮零 = 视角钝化、存疑）；确无新视角可换时，findings 集**跨一轮稳定为空**才算收敛。

**视角数量**分两路，都不许自行判档：控制面资产 / 机制文档按下文「按对象类型伸缩」定；领域产物（PRD / 设计 / 用例 / 接入等有专属 reviewer 的）按第 1 步各自域 rubric 的视角菜单定。**要不要评**与**末轮换不换视角**见下节「默认重评制」。

## 流程什么时候算结束 · 改动后何时必须重新评审

**流程到用户验收通过才算结束。** 在那之前，用户的任何澄清、补充、纠错，**都还在本流程里**——不是新任务，也不是打补丁。收到澄清＝回到受影响的那一步接着走，走到 review 步就照常评审。

> 最常见的误判：走完 skill 编号流程的最后一格，就以为流程结束了，此后用户提的修改被当成"顺手改一下"。**"评审通过"只对被审的那一版有效**——产物改了，旧结论不自动继承。

### 判据：默认重评制

> **正式名（全仓唯一，两个）**：本节 =「默认重评制」，派单方必须给出的那项输入 =「**派单基线**」（同名小节，见下文）。下游（skill / reviewer / ADR / 脚本注释）引用时**分两层**：
>
> - **允许**：一行摘要（说清这一步大概是干嘛的）+ 指向 `docs/harness/adversarial-review.md`「<正式名>」的指针。做事的人在自己上下文里就该知道这步要干嘛，**不该为一句话跳到另一个文件**。
> - **禁止**：复制清单 / 枚举 / 阈值 / 判据分支 / 口径细节——这些只在本文一处，改了别处不用跟（细节复制必漂，ADR-0022:42）。
> - **判断标准**：本文这段改了，下游那行要不要跟着改？**要跟＝复制了细节（违规）；不用跟＝摘要（允许）。**
>
> **不许另造别名**。下游措辞也必须与本节的 **opt-out 语义**一致：默认重评；**不许出现"够格才重评 / 命中清单才重评"这类 opt-in 说法**。

**默认重评。末次对抗评审通过之后，产物只要有实质改动，一律重新过评审才能再交付。** **重评是缺省路径**——不需要先论证"这次改动够格重评"。免评是例外，例外说明一句就行（见下）。

**免评口子**：**不改变任何语义**的改动（典型例子、**不封闭**：错别字、纯格式、补链接、不改含义的措辞调整）。**只要改动改了任何一处含义、行为、约束、范围或结论，就不属于免评口子**——哪怕只有一行、只有一个字。

**要免评的人必须写一行免评理由留痕**：写明"改了什么 + 为什么不改变语义"，落点定死在 `tasks/todo.md` 当前任务的 Review 段（与下文验收凭据同处，便于 eval 评委取证）。没写就按缺省路径走（重评）。

**什么算"实质改动"，本文只定原则、不给清单。** 原则就是上面那句——**改变了任何含义 / 行为 / 约束 / 范围 / 结论**，即为实质改动。**这里不列举例**：harness 要管的项目与产出物类型各不相同（服务端 / 前端 / 数据管道 / 工具；需求文档 / 设计 / 用例 / 脚本 / 接入骨架），任何举例都会把某一类的口径带进通用资产（违 **rule-0015**：控制面资产必须通用、不掺项目或产物类型假设），且清单一旦存在，就会把"不在清单上"读成"大概没事"——而缺省路径本来就是重评。

**需要更具体的判据时，就近定、别回填到本文**：目前各产出型 skill 均未另定，一律按本节原则判，确有需要再就近补；项目专属口径落 `projects/<工程>/AGENTS.md`（就近生效）。

**判据是"有没有改变语义"，不是"感觉大不大"。** "重大 / 琐碎 / 顺手 / 应该没影响"这类形容词不构成判断——它们说的是改动给人的印象，不是改动的效果（本仓实证：一次被自判成"顺手"的改动走单视角过关，补跑多视角后挖出多个 major，含整块内容缺失与一处安全缺陷，详见 lessons 2026-07-15）。**说不清"为什么不改变语义"，就按实质改动走。**

### "用户验收通过"的锚

上面说"流程到用户验收通过才算结束"，**验收通过必须有用户明示的凭据**，否则算未验收、流程未结束：

- **凭据形式（唯一）**：用户明示通过的**原话 + 日期**记进 `tasks/todo.md` 当前任务的 Review 段。**没有这条 = 未验收。**
- **明确禁止的推定**：用户沉默、用户未反对、"用户没说什么"、用户转去问别的、流程走完了最后一格——**一律不得推定为验收通过**。
- 依据 rule-0009（验收断言必须锚定唯一、真实、产出方的证据）：验收的产出方是**用户**，所以凭据只能来自用户明示，不能由执行方自行判定。

### 派单基线

派单方**必须**在派单信息里显式给出二选一——"本次是首评"，或"上次评审结论 + 此后改了什么"。这项输入即「派单基线」。域 reviewer 开审前核它，**缺就先要求派单方补齐再开审**。核的是"缺没缺这项输入"（客观事实），不是"改没改"（主观判断）——仓里没有"上次评审通过的版本"锚点，reviewer 无从自行比对。

### 本节靠什么执行（有意不上机器闸）

**本节是上下文约束，不挂机器闸——这是设计选择，不是缺口。** harness 的机器闸（`scripts/hook-policy.sh` 等）编制是**防严重后果与越权**：泄密、危险命令、不可逆破坏。"改完该不该重评"不属于这一类——**没有严重后果，也不属于"反复改进流程仍然做不好"的事**，写进做事时一定会读到的地方就够了。

上下文承载在三处：
- **各产出型 skill 的硬规则 / 收尾自检**——做事时它就在上下文里；
- **部分域 reviewer 的工作步骤 0**（`hc-tech-design` / `hc-prd` / `hc-e2e` / `hc-api`，它们审的产物会被反复改）——开审前核「派单基线」，缺就要求派单方补齐；
- **收尾 eval（rule-0005）考题 010**——独立评委照清单查一遍，结论写进评审报告。

`make verify` 的各 audit 不检查评审时效，**也不打算检查**。

### 按对象类型伸缩

本段只定**控制面资产 / 机制文档**（规则、skill 本体、模板、配置——**且无对口领域 reviewer rubric** 的 generic 文档审）的视角数量；领域产物审走第 1 步域 rubric，见前文分两路。

审这类对象别套代码审的角度矩阵（语言陷阱 / efficiency 等必然空转）——就 2–3 个对口视角**封顶**：关联项遗漏（现成机制 = `hc-doc-sync-reviewer` 对照 doc-sync-checklist）、语义冲突 / 前后矛盾（跨文件口径对照）、必要时指针有效性。文档审的"深"不等于视角多，等于**对照面全**。

**收敛口径**：对口视角用尽、无新视角可换时，按"findings 跨一轮稳定为空"收敛（同第 5 步），或预留一个对口视角（如指针有效性）作末轮新视角。哪怕是实质改动也不 fan out 到十几个角度——**换视角不等于加视角数**。

## 两个运行时怎么执行同一套（双栈中性）

编排写成**中性指令**（"并行开 N 个视角挑、汇总去重、回改到过、末轮换视角"），两个运行时各用原生机制执行、产出同一形状：

- **Claude Code**：用 **Workflow 工具**——`parallel` / `pipeline` fan out `agent(..., {agentType:'<reviewer>'})` 多实例，`label`/`phase` 分组，回改循环。
- **Codex**：用**原生多 agent 编排**（`multi_agent_v2`，`[agents] max_threads` 并发上限）——主 agent（总监）并行 spawn 同名双栈 reviewer 子 agent、等结果、汇总。`codex exec --json` 亦可 headless 跑。

> 差别只在"谁保证 fan out"：Claude Workflow 是脚本确定性、Codex 是主 agent 模型驱动——**编排指令一致**，都不许退回单 reviewer 一遍过。

## reviewer 子 agent 的角色

reviewer 子 agent（`.claude/agents/hc-*-reviewer.md` + `.codex/agents/hc-*-reviewer.toml` 双栈）**持有该域完整 rubric = 视角菜单**；被派时可能是"N 个并行视角之一"（按分配视角重点挑）、也可能独挑全 rubric。reviewer **只评不改**，回结构化清单（位置 / 严重度 / 问题 / 证据 / 修法）。约束本体在各 reviewer 上下文，本文只定编排、不复制判据。

**视角菜单以 reviewer 自带 rubric 为准。派单方可以指定重点、可以加视角，但不能删减 rubric 里标了「固定必查」的项。** 这解释了为什么视角不能全靠派单方现场想：派单方没想到的角度就一直没人看——实测一条违 rule-0015 的判据表连过 7 轮评审、20 多个 reviewer 实例都没被指出来，因为那 7 轮的视角全出自同一个派单方。把这类角度固定在 reviewer 自己的定义里，换谁派都在。具体固定哪几项，见各 reviewer 的「固定必查」节，本文不复列。

## 边界

- 本文管两件事：review **步的编排**（怎么派、几个视角、怎么收敛）+ **生效范围**（流程何时才算结束、改动后何时必须重评——见上节，**各 skill 统一照此，不各自另定**）。**判据 / rubric** 归各 reviewer 子 agent；**review 步排在流程第几格**归各 skill 本体。
  - 「重评」专指**域 reviewer 的对抗评审**（第 1–5 步那套 fan out）：`make verify` 机检（查结构 / 链接 / 索引）与收尾 eval（rule-0005，整轮打分）是另两条独立防线，跑过任一条都不构成"已重评"（lessons 2026-06-30）。
  - 历史：本条原写作"流程位置归各 skill 本体"，被读成"何时（再）进 review 步也归各 skill"，而各 skill 一个都没写——两边互推、无人负责，实测导致"用户澄清后改了产物却不重评"（lessons 2026-07-15、2026-07-20）。故"何时重评"于 2026-07-20 收归本文统一规定。
- 云端多 agent 深审（`/code-review ultra`）是另一条**用户触发、计费**的路，不归本 pattern、agent 起不了。
